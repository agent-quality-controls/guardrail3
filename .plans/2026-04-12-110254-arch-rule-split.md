# Arch Mixed-Rule Split

## Goal

Remove the remaining mixed-lane `arch` rules so each package lane matches the actual contract:

- file-tree checks file-tree only
- config checks config contents only
- source checks source files only

The specific target is to split the current mixed `RS-ARCH-07` and `RS-ARCH-08` behavior into lane-pure package rules and lane-pure package inputs.

## Current flaw

The current package extraction still carries mixed facts across lane boundaries:

- `RS-ARCH-07` uses:
  - file-tree facts: sibling `.rs` count, sibling dir count, module depth
  - config facts: dependency count from `Cargo.toml`
- `RS-ARCH-08` uses:
  - source facts: `lib.rs` exports and `#[cfg(feature = ...)]`
  - config facts: `[features].all`, `[features].default`

That means:

- file-tree currently depends on config facts
- source currently depends on config facts
- the public lane inputs are oversized and lie about the lane boundary

## Desired end state

### File-tree lane

Own only:

- `RS-ARCH-01`
- `RS-ARCH-03`
- `RS-ARCH-07A` structural crate split threshold

`RS-ARCH-07A` checks only:

- sibling `.rs` count
- sibling dir count
- module depth

### Config lane

Own:

- `RS-ARCH-05`
- `RS-ARCH-06`
- `RS-ARCH-07B` dependency-count split threshold
- `RS-ARCH-08B` feature-table contract

`RS-ARCH-07B` checks only:

- direct dependency count

`RS-ARCH-08B` checks only:

- `all` exists
- `all` is non-empty
- `default` exists
- `default` includes `all`

### Source lane

Own:

- `RS-ARCH-02`
- `RS-ARCH-04`
- `RS-ARCH-08A` facade export gating
- `RS-ARCH-09`

`RS-ARCH-08A` checks only:

- pub exports are gated
- pub exports are not gated directly on `all`

## Rule-ID decision

Split the mixed rules in packages instead of keeping one mixed ID.

Reason:

- a single mixed ID forces mixed inputs
- mixed inputs defeat the lane boundary
- package rules should be honest about what they consume

Compatibility, if needed later, can be handled outside the lane packages.

## Approach

1. Shrink public lane inputs in `g3rs-arch-types`
   - remove oversized `G3RsArchCrateNode` use from lane boundaries
   - add lane-local public inputs for:
     - file-tree crate structure
     - config crate thresholds
     - config feature contract
     - source facade gating

2. Split `RS-ARCH-07`
   - move structural threshold logic into `g3rs-arch-file-tree-checks`
   - move dependency-count threshold logic into `g3rs-arch-config-checks`
   - delete the mixed `RS-ARCH-07` package rule

3. Split `RS-ARCH-08`
   - keep source export-gating logic in `g3rs-arch-source-checks`
   - move feature-table checks into `g3rs-arch-config-checks`
   - delete the mixed `RS-ARCH-08` package rule

4. Update `g3rs-arch-ingestion`
   - compute private rich facts once
   - fan out into lane-pure public inputs
   - keep rule behavior aligned with the old app code, but no lane gets mixed facts

5. Add failing tests first
   - prove that source no longer needs config facts
   - prove that file-tree no longer needs dependency counts
   - prove config owns the split checks
   - pin exact outputs for the new split rules

6. Re-run adversarial rule-distribution audit
   - compare every original app rule body against package lane placement
   - require zero mixed-lane findings before calling `arch` done

## Alternatives considered

### Keep `RS-ARCH-07` and `RS-ARCH-08` as single package rules

Rejected.

- would preserve mixed inputs
- would keep lane boundaries dishonest
- would keep repeating this classification problem in later families

### Move `RS-ARCH-07` entirely to config

Rejected.

- sibling/file depth thresholds are file-tree checks

### Move `RS-ARCH-08` entirely to config

Rejected.

- export-gating in `lib.rs` is a source check
