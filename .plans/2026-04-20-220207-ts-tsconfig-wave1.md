## Goal

Build the first real `ts/tsconfig` family for `g3ts`.

End state:

- `g3ts validate --family tsconfig` works
- the family is split into:
  - `tsconfig-json-parser`
  - `g3ts-tsconfig-types`
  - `g3ts-tsconfig-ingestion`
  - `g3ts-tsconfig-config-checks`
- wave 1 enforces:
  - root `tsconfig.base.json` exists
  - root `tsconfig.base.json` parses as real JSONC
  - local `tsconfig.json` files parse
  - local `tsconfig.json` files either extend the root base or carry the strict baseline inline
  - the current 12 strict booleans are enforced
- no write path

## Approach

1. Build a dedicated parser package for `tsconfig` files.
   - Use a Rust JSONC parser, not ad hoc comment stripping.
   - Return one passive document type with:
     - `raw: serde_json::Value`
     - `typed: TsconfigParseState`
   - Typed surface needs only the fields required for wave 1:
     - normalized `extends`
     - `compilerOptions`

2. Build `g3ts-tsconfig-types`.
   - Shared input type for config checks.
   - Config state should be file-content oriented:
     - missing
     - unreadable
     - parse error
     - parsed root base plus parsed local configs

3. Build `g3ts-tsconfig-ingestion`.
   - Use `g3-workspace-crawl`.
   - Discover:
     - root `tsconfig.base.json`
     - descendant `tsconfig.json` files except the root base itself
   - Parse once through the parser package.
   - Normalize only the facts needed by checks.

4. Build `g3ts-tsconfig-config-checks`.
   - One rule per file.
   - Wave 1 rule cut:
     - base exists
     - base parseable
     - local config parseable
     - local config inheritance-or-inline contract
     - strict booleans

5. Wire `SupportedFamily::Tsconfig` into `apps/guardrail3-ts`.

6. Validate against the external TS app roots:
   - `/Users/tartakovsky/Projects/websmasher/websmasher/apps/web`
   - `/Users/tartakovsky/Projects/websmasher/websmasher/apps/landing`

## Key Decisions

- Use a dedicated parser package.
  - Reason: JSONC parseability is a real file concern and should not live in ingestion.
- Use a Rust JSONC parser for wave 1.
  - Reason: wave 1 needs parseability plus a small typed field set, not full TypeScript compiler resolution.
- Support both string and array `extends`.
  - Reason: TypeScript 5.0 added multiple `extends` entries.
- Keep module/emit/runtime option policy out of wave 1.
  - Reason: apps and publishable packages legitimately differ there.

## Alternatives Considered

- Reuse `serde_json` with comment stripping.
  - Rejected: fake subset, brittle, and design drift from real `tsconfig` syntax.
- Shell out to `typescript` Node APIs.
  - Rejected for wave 1: stronger than needed for the current rule cut and adds a runtime dependency edge before the family proves its simpler boundary.
- Put parsing directly in ingestion.
  - Rejected: repeats the old grouped-checker boundary mistake.
