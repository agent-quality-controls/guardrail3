## Goal

Build `packages/ts/arch` wave 1 and wire it into `g3ts`.

End state:

- dedicated TS arch family packages:
  - `g3ts-arch-types`
  - `g3ts-arch-ingestion`
  - `g3ts-arch-config-checks`
  - `g3ts-arch-source-checks`
  - `g3ts-arch-file-tree-checks`
- `g3ts validate --family arch` works on one explicit target root
- family seam mirrors live Rust `arch` as closely as TS allows

## Approach

1. Mirror the live Rust package split.
   - Keep parser reuse narrow.
   - Keep family-owned typed inputs in `g3ts-arch-types`.
   - Keep parse/discovery in ingestion.
   - Keep rules pure in the three check lanes.
2. Keep ownership target-root scoped.
   - No repo-global ownership pass.
   - One validated path in, one target-root arch snapshot out.
3. Keep wave 1 narrow and real.
   - Config lane:
     - detect root manifest facade declarations
     - require declared facade entrypoints to exist
   - Source lane:
     - root and nested `index.ts` / `index.tsx` facade files must be facade-only
   - File-tree lane:
     - recursive structural split limits under source trees
4. Wire `SupportedFamily::Arch` into `g3ts`.
   - Add a TS structure runner group, mirroring the Rust app split.

## Key Decisions

- Use package-manifest facade linkage as the TS analogue of Rust crate facade presence.
  - Why: TS packages express public entrypoints through `package.json`, not `lib.rs`.

- Keep app/package classification out of wave 1.
  - Why: `apparch` should own app-internal layer semantics later.
  - Wave 1 only enforces generic package-architecture pressure that fits both apps and shared packages.

- Reuse Rust structural split thresholds for the first cut.
  - Why: they already encode the package-shape pressure we want.
