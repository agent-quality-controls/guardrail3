# Goal

Remove the invented behavior-replay architecture language from the current fixture migration work and make the rule explicit:

- owned Rust structs used as fixture output must derive `serde::Serialize`
- fixture verifiers should serialize those structs with `serde_json`
- fixture3 compares received output against approved output
- no hand-written per-family adapters, exporters, ingestion suites, replay suites, replay record maps, or canonical fact layers

# Correct Contract

The fixture contract has only these project-owned concepts:

- fixture: the input filesystem state
- verifier command: the executable command fixture3 runs
- received output: JSON produced by the verifier command
- approved output: JSON stored under `behavior/golden`
- diff: fixture3 comparison between received and approved output

`fixture3.yaml` uses a `suites` key because fixture3 owns that schema. That key is allowed. It must not be expanded into project architecture terms such as ingestion suite or replay suite.

# Serialization Rule

For any output built from types owned by this repo:

- add `serde::Serialize` to the owned structs and enums
- call `serde_json::to_value` or `serde_json::to_string_pretty`
- normalize only values that are inherently unstable, such as absolute paths, temporary directories, hashes generated by Rust build artifacts, and elapsed time
- keep normalization generic and outside rule-family logic

If `serde::Serialize` cannot be derived:

- document the exact type
- document the field that blocks serialization
- document why the type cannot be changed
- do not create a custom adapter unless this documented reason exists
