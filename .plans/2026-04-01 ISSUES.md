# Arch Family Issues — 2026-04-01

## Target rule set (renumbered by group)

**facade/** (01-04):
- **RS-ARCH-01** — Every crate must have a facade entry point (lib.rs or main.rs)
- **RS-ARCH-02** — lib.rs is facade-only: no implementation, no broad re-exports
- **RS-ARCH-03** — Module directories must have mod.rs (not foo.rs convention)
- **RS-ARCH-04** — mod.rs is facade-only: same as lib.rs rule

**dependency/** (05-06):
- **RS-ARCH-05** — No boundary crossing: dependencies can't resolve inside another crate's tree
- **RS-ARCH-06** — Non-child dependencies require `shared = true` flag

**complexity/** (07-08):
- **RS-ARCH-07** — Force crate split when complexity exceeds thresholds
- **RS-ARCH-08** — Facade exports must be feature-gated with named sub-features

---

## ISSUE-1: RS-ARCH-02 — migrate from RS-CODE-27 and tighten

RS-CODE-27 (facade-only lib.rs) enforces that lib.rs contains only re-exports
and type/const definitions. This is a crate architecture concern, not a code
quality concern. It should move to Arch as RS-ARCH-02.

Three fixes needed:

1. **Drop profile gate.** Currently fires only when `profile_name == "library"`.
   Profile names are irrelevant. Every crate has a facade (lib.rs). The rule
   should enforce on every crate's lib.rs unconditionally.

2. **Forbid broad re-exports.** `pub use engine;` or `pub use engine::*;` in
   lib.rs re-exports an entire internal crate/module, leaking deep internals
   through the facade. Only specific item re-exports should be allowed
   (`pub use engine::Engine`, `pub use engine::Config`). This is the
   complement to RS-ARCH-05 — together they make it impossible to reach
   internals either through direct Cargo dependencies or through deep `use`
   paths via the facade.

3. **Migrate to Arch family.** This is the rule that makes lib.rs the sole
   gatekeeper for external access. It's architecture, not code quality.

## ISSUE-2: Arch must not be path-gated

`is_packages_root()` in `arch/facts.rs` hardcodes a `packages/` path filter.
Arch must operate on every crate it's routed to, regardless of filesystem path.
No folder awareness. It sees a tree of Cargo.toml files and the relationships
between them. That's it.

## ISSUE-3: RS-ARCH-01 — wrongly scoped to split library roots

Old RS-ARCH-02 checks that split roots have `[workspace]` + `[package]` +
library. The actual universal rule is simpler: every crate must have a facade
entry point (lib.rs or main.rs). Not gated on split mode, not gated on being
a library. Every crate, unconditionally.

## ISSUE-4: Old RS-ARCH-03 — useless, replaced

Old RS-ARCH-03 fires in two cases:
1. Workspace declared but no member crates — but a workspace with just src/
   and no sub-crates is perfectly valid.
2. Escalation tripped but no members yet — but that's RS-ARCH-07's job. If
   you must split, that's already the error. A second rule for "you must split
   and haven't" is redundant.

Delete. The "every package must belong to a workspace" concern is already
topology's job (RS-TOPOLOGY-10, RS-TOPOLOGY-12). Not arch.

## ISSUE-5: RS-ARCH-07 — escalation filters on library crates only

`is_library` check skips binary-only crates. A binary crate can grow just as
complex as a library crate. The complexity thresholds and forced splitting
should apply to every crate regardless of whether it has lib.rs or main.rs.

Also: sibling .rs file threshold should be raised from 6 to 10 to account for
infrastructure files (lib.rs, mod.rs, facts, inputs, etc.) that every directory
naturally has.

## ISSUE-6: RS-ARCH-05 — wrong mechanism and scope

Current RS-ARCH-04 matches dependency package names against workspace member
names. This is wrong in two ways:

1. **Name matching is bypassable.** A dependency can alias the package name:
   `whatever = { path = "../my-lib/crates/engine" }` — the alias "whatever"
   won't match the internal member name "my-lib-engine". Rule misses it.

2. **Scoped to workspaces and packages/ path.** The rule should not know about
   workspaces. It applies to every crate.

### What the rule should actually do

**A dependency must not cross into another crate's internals.**

For every `path = "..."` dependency from source crate S to target crate T:
1. Resolve T's directory.
2. Walk up from T toward the project root. Check each intermediate directory
   for a Cargo.toml (a crate boundary). Directories without Cargo.toml are
   just folders — ignored.
3. If an intermediate crate boundary exists that S is NOT inside of, error.
   The dependency crosses that crate's boundary — must depend on that crate's
   facade instead.

### Examples

```
lib/
  Cargo.toml
  crates/
    types/
      Cargo.toml
      crates/
        order_types/
          Cargo.toml
        customer_types/
          Cargo.toml
    families/
      f1/
        Cargo.toml

consumer/
  Cargo.toml
```

- `consumer -> lib` — OK. Resolves to lib's root, through its facade.
- `consumer -> lib/crates/types` — FORBIDDEN. Crosses lib's boundary.
  Consumer is not inside lib. Must depend on lib instead.
- `f1 -> types` — OK. Both f1 and types are inside lib. No boundary
  crossed. Goes through types's facade.
- `f1 -> types/crates/order_types` — FORBIDDEN. Crosses types's boundary.
  f1 is not inside types. Must depend on types instead.

## ISSUE-7: RS-ARCH-08 — new rule: facade exports must be feature-gated

Every facade export in lib.rs must be behind a feature flag. Enforces:

1. Every `pub mod` / `pub use` in lib.rs must have `#[cfg(feature = "...")]`
2. Cargo.toml must have `[features]` with an `all` feature enabling everything
3. `default = ["all"]` so consumers get everything unless they explicitly opt out
4. `all` must NOT directly gate any `pub` items — it must only enable other
   named sub-features. This forces at least one level of meaningful feature
   grouping.

```toml
# Correct
[features]
default = ["all"]
all = ["types", "api", "events"]
types = []
api = ["types"]
events = ["types"]
```

```rust
// Correct — gated on named sub-features, not "all"
#[cfg(feature = "types")]
pub mod user_types;
#[cfg(feature = "api")]
pub mod handlers;
```

```rust
// Forbidden — "all" directly gating pub items
#[cfg(feature = "all")]
pub mod user_types;
```

Error message should instruct the agent to split the facade's public surface
into meaningful named features, re-export them through `all`, and expose them
separately for consumers who want fine-grained control.

## ISSUE-8: RS-ARCH-03 — new rule: module directories must have mod.rs

Same facade principle as lib.rs for crates, applied one level down. If a
directory is used as a module namespace, it must have a proper entry point.

The rule:
1. Find `mod foo;` or `pub mod foo;` declarations in source files.
2. Check if sibling directory `foo/` exists and contains .rs files.
3. If yes, `foo/mod.rs` must exist.
4. The `foo.rs`-alongside-`foo/` convention (Rust 2018+) is forbidden.
   Only `foo/mod.rs` is accepted. This keeps the module's facade entry
   point inside the module directory, not floating next to it.
5. Wiring files from that directory via `#[path = "foo/..."]` is forbidden
   by this rule — that's a bypass of the module facade.

Exceptions (not module namespaces):
- `tests/`, `examples/`, `benches/`, `src/bin/` — each .rs file is its own
  compilation unit, not a module tree.
- Fixture/data directories containing .rs files as test data.

Detection is deterministic: `mod` declaration + directory with .rs files =
must have mod.rs. No semantic analysis needed.

Note: RS-CODE-24 (path attr detection) stays in code family as the escape
hatch detector. This arch rule is the structural requirement that makes the
escape hatch unnecessary.

## ISSUE-9: RS-ARCH-04 — new rule: mod.rs must be facade-only

Same principle as RS-ARCH-02 (facade-only lib.rs) applied to mod.rs. A mod.rs
file is the module facade — it should only contain `pub mod`, `pub use`,
type/const definitions. No implementation logic.

RS-ARCH-02 checks lib.rs. RS-ARCH-04 checks mod.rs. One rule per thing.

## ISSUE-10: RS-ARCH-06 — new rule: non-child dependencies require shared flag

By default, a crate can only be depended on by its parent. Any other access —
sibling, descendant of sibling, anything — requires the target crate to be
explicitly marked `shared = true`.

One flag covers both sibling access and ancestor access (going up past parent
boundary). No separate rules needed.

### Mechanism

Crate declares itself as shared via Cargo.toml:

```toml
[package.metadata.guardrail3]
shared = true
```

### Rule behavior

For every dependency from source S to target T:
1. Is T a direct child of S? (S's Cargo.toml is a parent directory of T.)
   If yes, allowed — parent accessing own child. No flag needed.
2. Otherwise, check if T has `shared = true` in its metadata.
3. If not, error.

Default is `shared = false`. This means:
- A crate is private to its parent by default.
- Only explicitly shared crates can be depended on by anyone other than their
  parent.
- Both sibling access and ancestor access (going up past a parent boundary)
  are controlled by the same flag.

### Error message guidance

The error should explain:
- Dependencies on non-child crates are forbidden by default to prevent
  coupling and spaghetti.
- If this crate is genuinely a shared dependency (like common types, contracts,
  or utilities), mark it `shared = true` in its Cargo.toml metadata.
- Shared crates should ideally be named to reflect their shared nature (e.g.,
  `shared`, `common`, `contracts`, `types`).
- Carefully consider whether the crate should truly be shared. If not, the
  consuming crate should depend on the parent facade instead, which re-exports
  what's needed.

### Examples

```
root/
  crates/
    shared/     ← shared = true
    article/    ← shared = false
      crates/
        parse/  ← shared = false
        types/  ← shared = false
    breadcrumb/ ← shared = false
```

- `article (facade) → article/parse`: ✅ parent accessing own child
- `article → shared`: ✅ shared is marked shared
- `article/parse → shared`: ✅ shared is marked shared (ancestor access OK)
- `breadcrumb → shared`: ✅ shared is marked shared
- `article → breadcrumb`: ❌ breadcrumb is not shared
- `breadcrumb → article`: ❌ article is not shared
- `article/parse → article/types`: ❌ types is not shared (sibling within article)
- `article/parse → breadcrumb`: ❌ breadcrumb is not shared

## ISSUE-11: Arch needs no folder/path/profile concepts

Arch should know:
- Cargo.toml tree (crate locations, dependencies with paths)
- lib.rs / mod.rs content (facade analysis)
- Source code (for mod declarations and feature gate checks)

Arch should NOT know:
- Folder names (packages/, apps/, crates/)
- Profile names (library, app, etc.)
- Workspace membership semantics
- Family-specific structures (hex layers, topology)
