## Goal

Build a `ts/apparch` family that mirrors the live Rust `apparch` package shape and purpose, but over TypeScript workspace packages instead of Cargo crates. The family should enforce package-layer dependency rules and narrow public-surface rules for one explicit target workspace root.

End state:

- `g3ts` supports `--family apparch`
- package group exists:
  - `packages/ts/apparch/g3ts-apparch-types`
  - `packages/ts/apparch/g3ts-apparch-ingestion`
  - `packages/ts/apparch/g3ts-apparch-config-checks`
  - `packages/ts/apparch/g3ts-apparch-source-checks`
- no `file-tree-checks` package unless the live TS implementation proves a real need later
- `ts/apparch` validates package-layer dependency directions and selected source-level public-surface rules on a TS workspace root

## Approach

### 1. Mirror the live Rust `apparch` seam exactly where it is structural

Rust reference:

- `g3rs-apparch-types`
- `g3rs-apparch-ingestion`
- `g3rs-apparch-config-checks`
- `g3rs-apparch-source-checks`

Why:

- live Rust `apparch` has no file-tree lane
- it is a package graph and public-surface family, not a structural-facade family

TS should copy that shape first, then diverge only where TypeScript semantics require it.

### 2. Define TS package-layer facts in `g3ts-apparch-types`

The TS family contract should be the package-graph analogue of Rust crate/layer inputs.

Planned types:

- `G3TsApparchLayer`
  - `Types`
  - `Logic`
  - `IoInbound`
  - `IoOutbound`
- `G3TsApparchDependencyKind`
  - `Dependency`
  - `DevDependency`
  - maybe `PeerDependency`
  - maybe `OptionalDependency`
- `G3TsApparchPackage`
  - package name
  - package root rel dir
  - package manifest rel path
  - optional layer
- `G3TsApparchDependencyEdge`
  - from package manifest
  - to package manifest
  - dep name
  - dep kind
- `G3TsApparchExternalDependency`
  - package manifest
  - dep name
  - dep kind
- `G3TsApparchPolicyState`
  - if a dedicated TS policy file exists later, model it here
  - wave 1 can likely omit it
- `G3TsApparchPublicItem`
  - package manifest rel path
  - source rel path
  - item name
  - owner name optional
  - kind
- `G3TsApparchConfigChecksInput`
  - packages
  - dependency edges
  - external dependencies
- `G3TsApparchSourceChecksInput`
  - packages
  - public items

Key decision:

- keep family inputs narrow snapshots, not raw parser documents
- follow the current stronger TS pattern, not the looser older Rust habit of passing parser documents through family types

### 3. Build package-graph ingestion in `g3ts-apparch-ingestion`

This is the key adaptation from Rust.

Rust ingestion does:

- load workspace root
- discover member crates
- classify layers from crate paths
- collect internal dependency edges
- collect external dependencies
- collect patch/replace bypasses
- collect public source items from entrypoint-reachable modules

TS ingestion should do the TypeScript equivalent:

- identify the target workspace root from the explicit validate path
- read root `package.json`
- read `pnpm-workspace.yaml` if present and needed for package discovery
- discover descendant workspace packages by `package.json`
- exclude nested noise that is not part of the workspace package graph
- classify each package into a layer from its rel path
- collect internal dependency edges from manifest dependency tables
- collect external dependency declarations
- collect public source items from package public entrypoints

Files to add under ingestion runtime:

- `run/config.rs`
- `run/source.rs`
- `run/error.rs`
- `run/model.rs`
- `run/workspace.rs`
- `run/tests/...`

Key decision:

- derive package membership from the actual workspace root and package manifests
- do not guess package membership from arbitrary directories

### 4. TS layer classification

Wave 1 needs a concrete layer mapping, or the family will have nothing stable to enforce.

Proposed initial mapping:

- `packages/types/*` or `*/types/*` -> `Types`
- `packages/logic/*` or `*/logic/*` -> `Logic`
- `packages/io/inbound/*` or `*/io/inbound/*` -> `IoInbound`
- `packages/io/outbound/*` or `*/io/outbound/*` -> `IoOutbound`

For app-local workspaces, the equivalent package-root relative shape should be used.

Key decision:

- start with path-based layer classification, mirroring how Rust currently derives layers from crate placement
- do not invent a new TS-only config file just to label layers in wave 1

Alternative rejected for wave 1:

- explicit layer declarations in `package.json`
  - rejected because Rust does not require that, and it would add a new config surface before proving the family

### 5. Build `g3ts-apparch-config-checks` from live Rust rule categories

Rust config lanes currently enforce:

- types dependency direction
- logic dependency direction
- io/outbound dependency direction
- same-layer cycles
- dev dependency direction
- types purity
- logic purity
- patch/replace bypass

Wave 1 TS config checks should mirror the parts that directly translate:

- `TS-APPARCH-CONFIG-01`
  - types dependency direction
- `TS-APPARCH-CONFIG-02`
  - logic dependency direction
- `TS-APPARCH-CONFIG-03`
  - io/outbound dependency direction
- `TS-APPARCH-CONFIG-04`
  - same-layer cycles
- `TS-APPARCH-CONFIG-05`
  - dev dependency direction
- `TS-APPARCH-CONFIG-06`
  - types purity over external deps
- `TS-APPARCH-CONFIG-07`
  - logic purity over external deps

Potential TS-specific adaptation:

- there is no Cargo `patch` / `replace` analogue worth copying directly here
- if pnpm overrides become relevant, they likely belong in `ts/package` or `ts/topology`, not `ts/apparch`

Key decision:

- do not force Rust patch/replace parity where the TS ecosystem has no equivalent ownership surface

### 6. Build `g3ts-apparch-source-checks` for public-surface rules only

Rust source lane currently enforces:

- IO traits in types
- types public surface restrictions

Wave 1 TS source lane should stay narrow and translate the same intent:

- collect public items from package public entrypoints
- enforce that `types` packages only expose allowed kinds of public surface
- optionally enforce that io-facing abstract boundary types live in `types`, not `logic` or concrete IO packages

Concrete source rules to start:

- `TS-APPARCH-SOURCE-01`
  - `types` public surface only exposes allowed declarations
- `TS-APPARCH-SOURCE-02`
  - inbound/outbound abstract contracts that are intentionally public must be owned from `types`

Key decision:

- source lane is about public entrypoint surface, not whole-package AST crawling
- follow the Rust pattern of entrypoint-reachable source collection

### 7. Wire `ts/apparch` into `g3ts`

App changes:

- add `SupportedFamily::Apparch`
- add a structure-oriented family runner entry for `apparch`
- reuse the same runner group as `arch` unless the implementation proves it needs a separate group

Key decision:

- `apparch` belongs with structure families, not config-policy families

### 8. Verification and attack pass

Before reporting done:

- unit tests prove:
  - package discovery
  - layer classification
  - internal dependency edge extraction
  - external dependency extraction
  - cycle detection
  - entrypoint public-item collection
  - exact rule payloads for first rule wave
- `cargo test` passes for all new roots
- `g3rs validate` passes for all new roots
- `g3rs validate --path apps/guardrail3-ts` stays clean
- one adversarial review compares the implementation against:
  - live Rust `apparch` package shape
  - this plan

## Key decisions

- Build `ts/apparch` next, not `ts/topology`.
  - Why: the next pain point is package-layer dependency architecture, which matches Rust `apparch`.
  - Rejected: `ts/topology` first, because that would solve workspace-root attachment problems, not package-layer dependency direction.

- No `file-tree-checks` lane in wave 1.
  - Why: live Rust `apparch` does not have one.
  - Rejected: copying `arch` shape into `apparch` just for symmetry.

- Package/workspace discovery should be rooted in actual TS workspace files.
  - Why: `apparch` is a package graph family.
  - Rejected: arbitrary directory walking without workspace membership grounding.

- Use path-derived layer classification first.
  - Why: it mirrors the current Rust style and avoids a new config surface.
  - Rejected: explicit layer metadata in manifests for wave 1.

## Explicit non-goals for wave 1

- no `ts/topology`
- no repo-global ownership pass
- no source-module import graph enforcement inside one package
- no TS code-generation or write path
- no manifest-policy duplication already owned by `ts/package`
