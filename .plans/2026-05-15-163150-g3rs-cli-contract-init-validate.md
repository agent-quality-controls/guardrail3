# G3RS CLI Contract: Init And Validate

## Goal

Define the new CLI contract before implementation.

The command model is:

```text
g3rs init repo
g3rs init workspace --path <path>
g3rs validate repo
g3rs validate workspace --path <path>
```

There are no compatibility aliases.

Remove these commands:

```text
g3rs validate-repo
g3rs validate --path <path>
```

The CLI must teach the architecture from root help because agents often only run:

```text
g3rs --help
```

## Product Contract

The command nouns must mean one thing.

- `init` writes files or Git config.
- `validate` reads files and reports findings.
- `repo` means the Git repository control surface containing `--path`.
- `workspace` means one adopted Rust workspace or package root at `--path`.

The tool must not blur these boundaries.

- `validate repo` must not install hooks.
- `validate workspace` must not create guardrail files.
- `init repo` must not create Rust workspaces.
- `init workspace` must not touch Git hook config.

## Root Help Contract

Root help is the primary agent-facing documentation surface.

The root help must front-load:

- what a repo is
- what a workspace is
- the correct first commands
- the difference between init and validate
- the fact that old command shapes do not exist
- the hook lifecycle
- the workspace lifecycle

Root help should be hand-authored with Clap `after_help` or equivalent, not left as default subcommand listing only.

Required root help content:

```text
G3RS enforces Rust repo setup and Rust workspace guardrails.

Start here:
  g3rs init repo
  g3rs init workspace --path <path>
  g3rs validate repo
  g3rs validate workspace --path <path>

Concepts:
  repo       Git repository surface: hooks, repo-level topology, marker pairs.
  workspace  One adopted Rust unit: Cargo.toml plus guardrail3-rs.toml.

Rules:
  init writes setup.
  validate only reports.
  validate repo checks that Git will run G3RS.
  validate workspace checks one Rust unit.
```

The exact text can be edited, but these facts must appear in root help, not only in subcommand help.

## Command Shape

### `g3rs init repo`

Purpose:

- Bootstrap repo-level guardrails.

Arguments:

- `--path <path>` optional. Defaults to `.`.
- `--force` optional.

Behavior:

- Resolve the Git repo root containing `--path`.
- Create `.githooks/`.
- Create `.githooks/pre-commit.d/`.
- Create or update `.githooks/pre-commit.d/g3rs`.
- Create `.githooks/pre-commit` as a thin adapter that runs `.githooks/pre-commit.d/g3rs` when no hook exists.
- Integrate a bounded adapter block into `.githooks/pre-commit` when a project-owned hook already exists.
- Set `git config core.hooksPath .githooks`.
- Set executable bits on generated hook scripts.
- Apply the force contract below.
- Run `g3rs validate repo --path <path>` after writing.

## Existing Hook Adapter Contract

Some repositories already have a pre-commit adapter. `g3rs init repo` must support that case.

There is still only one command:

```text
g3rs init repo --path <path> [--force]
```

It handles three hook states.

### No Hook Exists

`g3rs init repo` creates a managed `.githooks/pre-commit` adapter and a managed `.githooks/pre-commit.d/g3rs` script.

Git only runs `.githooks/pre-commit`. It does not know about `.githooks/pre-commit.d/`. The adapter makes the one managed G3RS file real.

### Managed Hook Exists

`g3rs init repo` updates the managed adapter, managed block, and managed G3RS script.

Managed files are recognized only by stable markers, not by vague text search.

### Project-Owned Hook Exists

When `.githooks/pre-commit` exists and is not generated by G3RS:

- without `--force`: refuse to modify it
- with `--force`: insert or update one bounded G3RS managed block

The managed block is the only content `g3rs` may insert into a project-owned hook.

Example block:

```sh
# g3rs:begin managed repo guardrails
repo_root="$(git rev-parse --show-toplevel)"
g3rs_hook="$repo_root/.githooks/pre-commit.d/g3rs"
if [ -x "$g3rs_hook" ]; then
    "$g3rs_hook"
else
    echo "ERROR: missing executable $g3rs_hook"
    exit 1
fi
# g3rs:end managed repo guardrails
```

Insertion rules:

- Insert after the shebang and immediate shell option lines such as `set -e` or `set -eu`.
- If there is no shebang, insert at the top only when the file is clearly shell syntax.
- If the hook uses an unsupported interpreter, refuse.
- If an existing G3RS managed block is present, replace only the block contents.
- If more than one G3RS managed block is present, refuse.
- If the file is not valid UTF-8 text, refuse.
- Preserve all project-owned lines outside the managed block byte-for-byte.
- Do not reorder, delete, or rewrite project checks.

This is insertion, not replacement.

The managed `.githooks/pre-commit.d/g3rs` script owns the real G3RS logic:

- `g3rs validate repo --path "$repo_root"`
- discovery of adopted Rust workspaces
- `g3rs validate workspace --path "$unit" --staged`

Hook validation must prove this exact chain:

- Git runs `.githooks/pre-commit`.
- `.githooks/pre-commit` runs `.githooks/pre-commit.d/g3rs`.
- `.githooks/pre-commit.d/g3rs` exists.
- `.githooks/pre-commit.d/g3rs` is executable.
- `.githooks/pre-commit.d/g3rs` contains the required G3RS commands.

This is not a general `pre-commit.d` framework. Guardrails do not need to prove that every `.githooks/pre-commit.d/*` file runs. They only need to prove that the managed G3RS file runs.

### `g3rs init workspace --path <path>`

Purpose:

- Bootstrap one Rust workspace or package root.

Arguments:

- `--path <path>` required.
- `--force` optional.

Behavior:

- Require `Cargo.toml` at `--path`.
- Create `guardrail3-rs.toml` if missing.
- Create required Rust policy files if missing.
- Apply the force contract below.
- Run `g3rs validate workspace --path <path>` after writing.

No CLI `--profile` exists in this contract.

`guardrail3-rs.toml` currently has a parsed `profile` field with `service` and `library` values. That is existing config data, not a new init command axis. `init workspace` may write the documented default config value only after the project-shape docs define it.

### `g3rs validate repo`

Purpose:

- Validate repo-level guardrails.

Arguments:

- `--path <path>` optional. Defaults to `.`.
- `--inventory` optional.

Behavior:

- Resolve the Git repo root containing `--path`.
- Run hooks family.
- Run repo-level topology checks.
- Run marker-pair checks.
- Fail if Git is not configured to run G3RS.

This command replaces `g3rs validate-repo`.

### `g3rs validate workspace --path <path>`

Purpose:

- Validate one Rust unit.

Arguments:

- `--path <path>` required.
- `--family <family>` repeatable.
- `--inventory` optional.
- `--staged` optional.
- `--rules-only` optional.

Behavior:

- Require `Cargo.toml` at `--path`.
- Require or validate `guardrail3-rs.toml` at `--path`.
- Run selected workspace families.
- Run cargo gates unless `--rules-only` is set.
- Apply `--staged` only inside this workspace.

This command replaces `g3rs validate --path <path>`.

## Type Contract

Update app command types so command structure mirrors product structure.

Current type:

```rust
pub enum Command {
    Validate { ... },
    ValidateRepo { ... },
}
```

Target type:

```rust
pub enum Command {
    Init(InitCommand),
    Validate(ValidateCommand),
}

pub enum InitCommand {
    Repo(InitRepoRequest),
    Workspace(InitWorkspaceRequest),
}

pub enum ValidateCommand {
    Repo(ValidateRepoRequest),
    Workspace(ValidateWorkspaceRequest),
}
```

Rename request types:

```rust
ValidateRequest -> ValidateWorkspaceRequest
ValidateRepoRequest stays but moves under ValidateCommand::Repo
```

Add request types:

```rust
InitRepoRequest {
    path: PathBuf,
    force: bool,
}

InitWorkspaceRequest {
    workspace_root: PathBuf,
    force: bool,
}
```

Do not keep parsing support for deleted command shapes.

Tests must assert that these fail to parse:

```text
g3rs validate-repo
g3rs validate --path .
```

## Output Contract

Output must be useful when an agent only sees stdout and stderr.

### Init Output

Successful init output must say what changed and what validation command ran.

Example:

```text
initialized repo
created .githooks/pre-commit
created .githooks/pre-commit.d/g3rs
configured core.hooksPath=.githooks
validated with: g3rs validate repo --path <path>
```

If nothing changed:

```text
repo already initialized
validated with: g3rs validate repo --path <path>
```

On refusal:

```text
refused to modify .githooks/pre-commit
reason: project-owned hook needs explicit managed-block insertion
rerun with --force to insert the G3RS managed block
```

## Force Contract

`--force` must have a precise meaning.

Non-force behavior:

- Create missing directories.
- Create missing files.
- Leave byte-identical managed files unchanged.
- Refuse to modify an existing generated file whose content differs.
- Refuse to modify a project-owned file.
- Refuse to modify `guardrail3-rs.toml` if required changes would remove existing keys, waivers, or allowlists.
- Print every refusal with the file path and reason.

Force behavior:

- Replace only G3RS-owned generated files as whole files.
- Insert or update the G3RS adapter block inside a project-owned hook.
- Update `core.hooksPath` to `.githooks`.
- Do not delete unrelated files.
- Do not delete `.githooks/pre-commit.d/*` scripts.
- Do not replace an entire project-owned hook script.
- Do not replace `guardrail3-rs.toml` as a whole file.

Managed generated files:

- `.githooks/pre-commit.d/g3rs`

Managed generated files must contain a stable header:

```text
# Generated by g3rs. Do not edit this file directly.
```

If an existing managed generated file differs, `--force` rewrites the whole file atomically.

If `.githooks/pre-commit` exists and lacks the generated header or managed block:

- without `--force`: refuse
- with `--force`: insert the managed G3RS block when the hook is a supported shell script

Reason: replacing a project-owned hook can silently delete project checks. The supported force action is bounded insertion, not destructive replacement.

Structured config files:

- `guardrail3-rs.toml` is not a whole-file generated artifact once it exists.
- Init must parse it and apply a structured update only for documented missing keys.
- Init must preserve waivers, `allowed_deps`, unknown keys, and comments if the chosen TOML edit library can preserve comments.
- If comments cannot be preserved, init must refuse rather than rewrite the file.
- `--force` does not permit lossy TOML rewrites.

### Validate Output

Validation output must keep the current finding shape but add command scope at the top.

Example:

```text
scope: repo
root: /path/to/repo

== hooks ==
[Error] g3rs-hooks/pre-commit-exists .githooks/pre-commit pre-commit hook missing
  Expected a cached `.githooks/pre-commit` or `hooks/pre-commit` hook.
```

Workspace validation:

```text
scope: workspace
root: /path/to/workspace

No findings.
```

The renderer can remain plain text. It must include scope and root because agents need to know whether they validated the repo or a workspace.

## Key Decisions

- No compatibility aliases.
- Root help is a deliverable, not generated filler.
- The command tree mirrors the architecture.
- Init and validate are separate execution paths.
- Scope and root appear in validation output.
- Deleted command forms must fail in tests.
- `--path` is the only location flag.
- `--force` never permits lossy TOML rewrites.
- Existing project hooks are integrated by bounded managed-block insertion.
