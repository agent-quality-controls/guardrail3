# All Rust Ingestion Serialized Fixtures

## Goal

Replace the remaining ingestion unit-test behavior with fixture3 golden output by serializing each family ingestion output directly from Rust-owned structs.

The finished state is:

- every Rust ingestion package has a `crates/fixture-output` binary crate
- every binary accepts `--path <workspace-root>`
- every binary runs the package's public ingestion entry points against `g3_workspace_crawl::crawl`
- every binary prints JSON generated by `serde_json`
- every serialized output is the real owned ingestion output or real owned ingestion error
- Python fixture scripts only run the binary, normalize paths, and wrap fixture3 records
- no Python adapter, field mapping, hand-selected replay record, or family-specific JSON reconstruction exists
- every `needs_serialized_ingestion_output` ledger row is either covered by a fixture3 golden record or remains kept with a specific reason

## Why Code Still Has 49 Remaining Rows

`g3rs-code-ingestion` has serialization infrastructure, but not enough fixture coverage.

- The successful code-ingestion record currently serializes:
  - `config_checks.Ok.files`: 3 files

  - `file_tree_checks.Ok.roots`: 1 root
- The 49 remaining code rows cover more branches than those two fixtures expose:
  - custom library and binary target classification
  - default library and binary root classification
  - nested workspace member ownership
  - virtual workspace non-ownership
  - shared crate metadata
  - waiver threading from root and package configs
  - exception comments across supported config files
  - unsafe-code lint extraction from both compact and detailed Cargo forms
  - unreadable owned config/source failures
  - malformed nearest Cargo failures

Conclusion:

- The previous code-ingestion serialization work was structurally useful.

## Architecture

### Fixture-Output Crate Shape

Each ingestion package gets one binary crate:

```text
packages/rs/<family>/g3rs-<family>-ingestion/crates/fixture-output
```

Each crate must expose:

- `run_from_env() -> Result<(), FixtureOutputError>`
- `render_path(path: &Path) -> Result<String, FixtureOutputError>`
- `FixtureOutputError`

Each crate must contain:

- `src/lib.rs`
- `src/main.rs`
- `src/run.rs`

No implementation goes in `lib.rs` except module wiring and public exports.

### Serialized Output Shape

Each fixture-output crate prints one JSON object:

```json
{
  "schema_version": "g3rs-<family>-ingestion-fixture-output-v1",
  "config_checks": { "Ok": ... } or { "Err": ... },
  "source_checks": { "Ok": ... } or { "Err": ... },
  "file_tree_checks": { "Ok": ... } or { "Err": ... }
}
```

Families with extra lanes include those lanes by their exact public ingestion name.

Example:

- `g3rs-release-ingestion` also emits `repo_root_checks` because it exports `ingest_for_repo_root_checks`.
- `g3rs-topology-ingestion` emits only `file_tree_checks` because it only exports `ingest_for_file_tree_checks`.

### Serialization Rule

Every serialized field must come from a Rust type owned by the family packages.

Allowed:

- `#[derive(serde::Serialize)]` on owned input structs
- `#[derive(serde::Serialize)]` on owned error enums
- `#[serde(...)]` attributes only when needed for stable enum/tag output

Forbidden:

- Python adapters
- Rust adapters that copy fields into parallel output structs
- hand-written `to_json` methods
- fixture-only mirror structs when the real type is owned by this repo
- selecting a subset of fields because the full type is inconvenient

If `Serialize` cannot be derived for a real type, document the exact blocker in the plan/worklog before adding any workaround.

### Python Script Shape

Each family gets one fixture3 script:

```text
scripts/behavior/fixture3-g3rs-<family>-ingestion.py
```

The script may:

- copy the fixture repo to a temp directory
- run the family fixture-output binary
- parse stdout as JSON to fail on invalid JSON
- normalize absolute paths to deterministic placeholders
- emit the fixture3 envelope

The script must not:

- interpret Rust semantics
- map Rust fields manually
- drop fields
- rename family-specific fields
- decide which findings matter

## Fixture Design Rule

Do not create one fixture per unit test.

For each family:

1. Read the kept ledger rows for that family.
2. Group rows by the lowest behavior layer that can expose them.
3. Merge rows into one fixture when the same fixture can expose all of them without preventing any later row from firing.
4. Split fixtures only when one broken condition prevents ingestion from reaching another condition.
5. Prefer one dense fixture per family lane:
   - one clean branch-dense fixture
   - one malformed parse fixture if parse failure stops the lane
   - one unreadable file fixture if filesystem failure stops the lane
   - one foreign/nested workspace fixture when ownership behavior is different from normal workspace behavior

No ledger row can be marked replaceable until a fixture3 approved record contains the serialized behavior that would fail if that row regressed.

- one nested-workspace ownership fixture covering:
  - nested workspace members from their own manifest
  - foreign nested repository config ignoring
  - virtual workspace root not owning `src/lib.rs`
  - unowned source with no profile
- one metadata and waiver fixture covering:
  - `package.metadata.guardrail3.shared = true`
  - normal crate with shared flag false
  - root `guardrail3-rs.toml` waivers
  - package-root `guardrail3-rs.toml` waivers
- one config-extraction fixture covering:
  - exception comments in every supported owned config filename
  - multiple exception comments with exact lines
  - parser-backed config selection only
  - `workspace.lints.rust.unsafe_code` compact form
  - detailed unsafe-code lint form
  - harmless comments and quoted false positives
- one file-tree workspace fixture covering:
  - glob workspace members
  - workspace excludes
  - owned roots with no Rust files
- one source-pipeline fixture covering:
  - TODO macro findings
  - panic findings
  - direct filesystem API findings
  - allow-attribute findings
  - garde skip findings
  - line-count and dispatch-boundary findings
  - trait and public error boundary findings
  - `include_str!` traversal findings
  - known false-positive patterns that must remain clean
  - parse failure in one source file while other findings still emit
- one malformed/unreadable fixture per stop-the-lane failure:
  - malformed owned root `Cargo.toml`
  - malformed nearest `Cargo.toml`
  - unreadable owned config file
  - unreadable selected source file

After those fixtures pass, update only the code-ingestion ledger rows proven by the approved output.

2. `bash scripts/behavior/verify-all.sh`.
3. `g3rs validate repo --path "$PWD"`.

## Deletion Rule

- the fixture3 approved output exists
- the ledger row disposition changes from `needs_serialized_ingestion_output` to a replaceable disposition
- `python3 scripts/behavior/verify-test-deletion.py` passes
- deleting the test file does not remove tests that still have non-replaceable rows
