# Generate and Dry-Run Flow for Complex Monorepos

**Date:** 2026-03-18
**Scope:** Design document for `generate`, `diff`, `check`, and related commands in monorepo contexts

---

## 1. The Ownership Model: Generated vs Validated vs User-Managed

Every config file guardrail3 touches falls into one of three categories. This classification is the foundation of every decision that follows.

### Generated (guardrail3 owns the file)

guardrail3 produces the complete file from modules + overrides. The user never edits it directly -- they add overrides in `.guardrail3/overrides/` and guardrail3 regenerates.

| File | Scope | Notes |
|------|-------|-------|
| `clippy.toml` | Per-app + root packages | Composed from method/type ban modules + overrides |
| `deny.toml` | Per-app + root packages | Composed from section modules + overrides |
| `rustfmt.toml` | Per-app + root packages | Canonical, no overrides needed |
| `rust-toolchain.toml` | Project root | Canonical |
| `release-plz.toml` | Project root | Service profile only |
| `cliff.toml` | Project root | Service profile only |
| `.githooks/pre-commit` | Project root | Profile-aware builder |
| `.npmrc` | Project root | Canonical |
| `tsconfig.base.json` | Project root | Canonical strict base |
| `.jscpd.json` | Project root | Canonical |
| `cspell.json` | Project root | Generated with custom word merge |

**Marker:** Every generated file starts with a header comment:
```toml
# Generated by guardrail3 — DO NOT EDIT DIRECTLY
# Customize via .guardrail3/overrides/ or guardrail3.toml
# Re-generate: guardrail3 generate
```
For JSON files (no comments), use a `"_guardrail3": "generated"` key. For `.mjs` files, use a `// Generated by guardrail3` banner.

The marker serves three purposes:
1. `generate` can detect "this is mine, safe to overwrite"
2. `diff` can distinguish "guardrail3-managed file has drifted" from "user's file I shouldn't touch"
3. Humans can see at a glance whether they should edit a file or its overrides

### Validated (user owns the file, guardrail3 checks it)

The user writes these files. guardrail3 does NOT generate them, but `validate` checks that required settings/rules/plugins are present.

| File | What guardrail3 validates |
|------|--------------------------|
| `Cargo.toml` `[workspace.lints]` | All required clippy lints present at correct severity (R26-R29) |
| Per-app `tsconfig.json` | `strict: true`, extends `tsconfig.base.json` (T9-T10, T52-T54) |
| Root `eslint.config.mjs` | Required plugins configured, key rules present (T1-T8, T36-T51) |
| Per-app `eslint.config.mjs` | Required plugins for app type (T60+) |
| Per-app `package.json` | Required devDependencies, no banned deps, correct scripts (T15-T18, T55-T58) |
| `.stylelintrc.mjs` | Required notation rules for content apps |

These files are never touched by `generate`. They are only read by `validate`.

### Gray Zone Resolution

**Root `eslint.config.mjs`:** This is the hardest case. The real steady-parent file is 487 lines of project-specific rules, import groups, boundary configs. guardrail3 cannot generate this -- it would destroy the user's work. But it can and should validate that required plugins and rules are present.

**Decision: `eslint.config.mjs` moves to Validated.** guardrail3 removes it from the generate set. Instead:
- `validate` checks: are required ESLint plugins configured? Are key rules present? (Already implemented as T1-T8, T36-T51)
- For greenfield projects: `rs init` / `ts init` can scaffold a STARTER eslint config. But once the user edits it, guardrail3 never overwrites it again.
- The starter lives in `show-module canonical/eslint-starter` for reference.

**`.stylelintrc.mjs`:** Same treatment -- move to Validated. Custom CSS notation rules are project-specific. `validate` can check that required rules exist.

**`cspell.json`:** Gray zone because the user adds custom words but the structure is generated. **Decision: Generated with merge.** guardrail3 generates the base config and merges the user's `words` array from the existing file. This is safe because the `words` array is append-only and order-independent.

### Config Classification in `guardrail3.toml`

The user can override the default classification:

```toml
[typescript.canonical]
npmrc = true              # generate (default: true)
tsconfig_base = true      # generate (default: true)
jscpd = true              # generate (default: true)
eslint = "validate"       # validate only, don't generate (default: "validate")
stylelint = "validate"    # validate only (default: "validate" for content apps)
cspell = "generate"       # generate with word merge (default: "generate")
```

When `eslint = "generate"` is explicitly set, guardrail3 generates a starter config (useful for greenfield). But if an existing eslint config WITHOUT the guardrail3 marker is detected, it refuses with a clear message.

## 2. Per-App Override Architecture

### The Problem

In steady-parent, `apps/validator-rust/clippy.toml` has custom method bans:
```toml
{ path = "reqwest::Client::builder", reason = "Use shared clients" },
{ path = "reqwest::Client::new", reason = "Use shared clients" },
{ path = "std::env::var", reason = "Use config module" },
```

And `apps/substack-publisher/clippy.toml` has DIFFERENT custom bans. The current `.guardrail3/overrides/` system is global -- one set of overrides for all apps. This doesn't work.

### Solution: Per-App Overrides

```
.guardrail3/
  overrides/                     # Global overrides (applied to all apps)
    clippy-methods.toml
    clippy-types.toml
    deny-bans.toml
    deny-skip.toml
    deny-feature-bans.toml
  overrides/apps/                # Per-app overrides
    validator-rust/
      clippy-methods.toml        # Extra methods banned only in validator-rust
      clippy-types.toml
      deny-bans.toml             # Extra crate bans (e.g., deny anyhow)
      deny-skip.toml
    substack-publisher/
      clippy-methods.toml        # Different extra methods
      deny-bans.toml
```

**Resolution order:**
1. Start with profile baseline (e.g., `service` methods + types)
2. Apply global overrides (`.guardrail3/overrides/clippy-methods.toml`)
3. Apply per-app overrides (`.guardrail3/overrides/apps/{name}/clippy-methods.toml`)
4. Deduplicate entries already in the baseline

This means `generate_rust_files` changes from:
```
for each app:
  build clippy.toml(profile, is_pure, global_overrides)
```
to:
```
for each app:
  per_app_overrides = load_per_app_overrides(app_name)
  merged = merge(global_overrides, per_app_overrides)
  build clippy.toml(profile, is_pure, merged)
```

### Root Packages

Root-level `clippy.toml` / `deny.toml` (for `packages/`) use only the global overrides, not any per-app overrides. The root configs represent the library baseline.

## 3. The Generate Flow

### Step-by-step

```
guardrail3 generate [--dry-run] [path]
guardrail3 rs generate [--dry-run] [path]
guardrail3 ts generate [--dry-run] [path]
```

**Phase 1: Load and resolve**
1. Read `guardrail3.toml`
2. Determine profile(s)
3. Discover workspace structure (Cargo.toml members, package.json workspaces)
4. Resolve app names to filesystem paths (e.g., `validator-rust` -> `apps/validator-rust/`)
5. Load global overrides from `.guardrail3/overrides/`
6. Load per-app overrides from `.guardrail3/overrides/apps/{name}/`

**Phase 2: Generate content (in memory)**
7. For each app: compose clippy.toml, deny.toml, rustfmt.toml from profile + global overrides + per-app overrides
8. For root packages (if applicable): compose from library profile + global overrides
9. For project root: rust-toolchain.toml, release-plz.toml, cliff.toml
10. For TS: .npmrc, tsconfig.base.json, .jscpd.json, cspell.json (with word merge)
11. Hooks: .githooks/pre-commit

**Phase 3: Classify each file**
12. For each generated file, compare against disk:
    - **Would create:** File doesn't exist
    - **No changes:** File exists, content matches exactly
    - **Would update (managed):** File exists, has guardrail3 marker, content differs
    - **Would update (has custom content):** File exists, has guardrail3 marker, content differs, AND existing file has entries not in the new generated content (these need extracting to overrides)
    - **CONFLICT (unmanaged):** File exists, does NOT have guardrail3 marker -- guardrail3 did not create this file

**Phase 4: Act (or report if --dry-run)**

If `--dry-run`: print the report (see section 4) and exit with code 1 if changes pending, 0 if none.

If not `--dry-run`:
- **Would create:** Write the file with marker header
- **Would update (managed):** Overwrite the file
- **Would update (has custom content):** This is the critical case. See "Custom Content Handling" below.
- **CONFLICT (unmanaged):** Refuse to overwrite. Print a clear error explaining the options.

### Custom Content Handling (the hard problem)

When `generate` detects that a guardrail3-managed file has entries that are NOT in the new generated content, these are user customizations that were added directly to the file instead of through overrides.

**Behavior: Extract and regenerate.**

1. Extract the custom entries from the existing file
2. Write them to the appropriate override file (`.guardrail3/overrides/apps/{name}/clippy-methods.toml`)
3. Regenerate the file with those overrides now included
4. Print what was extracted and where

Example output:
```
  apps/validator-rust/clippy.toml:
    Extracted 3 custom method bans to .guardrail3/overrides/apps/validator-rust/clippy-methods.toml:
      reqwest::Client::builder — "Use shared clients"
      reqwest::Client::new — "Use shared clients"
      std::env::var — "Use config module"
    Regenerated with overrides applied.
```

This is non-destructive: the custom entries survive in overrides, and the next `generate` includes them automatically.

**Edge case: What if the extraction is wrong?** The entry detection is TOML-structural (`{ path = ... }` and `{ name = ... }` patterns). If the user has added non-standard TOML content that doesn't match these patterns, it would be lost. Mitigation:
- Before any destructive operation, write a `.guardrail3/backup/{filename}.{timestamp}` backup
- Print a warning: "Backed up original to .guardrail3/backup/..."
- The backup is gitignored (add to `.guardrail3/.gitignore`)

### Conflict Resolution (unmanaged files)

When `generate` finds a file without the guardrail3 marker:

```
CONFLICT: apps/validator-rust/clippy.toml exists but is not managed by guardrail3.

Options:
  1. guardrail3 adopt apps/validator-rust/clippy.toml
     — Extracts custom entries to overrides, marks file as managed
  2. guardrail3 generate --force
     — Overwrites all files (DESTRUCTIVE — custom entries not in overrides will be lost)
  3. Add marker manually:
     Add "# Generated by guardrail3" as the first line, then re-run generate
```

The `adopt` command (new) handles the common case: the user has an existing clippy.toml they want guardrail3 to manage going forward.

## 4. Dry-Run Output Design

### Principles

1. **Group by location, not by status.** The user thinks in terms of "what happens to validator-rust?" not "which files would be created?"
2. **Show custom entries explicitly.** The user needs to know guardrail3 detected their customizations and will preserve them.
3. **Differentiate managed vs unmanaged.** Conflicts (unmanaged files) need to stand out clearly.
4. **Summary at the end.** Quick scan: how many changes, how many conflicts, how many custom entries.

### Output Format

```
guardrail3 rs generate --dry-run

Rust configs:

  apps/validator-rust/
    clippy.toml .......... update (28 lines differ)
      Custom entries detected (3 — would extract to overrides):
        methods: reqwest::Client::builder — "Use shared clients"
        methods: reqwest::Client::new — "Use shared clients"
        methods: std::env::var — "Use config module"
      Generated changes:
        + 3 new method bans added to baseline
        ~ cognitive-complexity threshold: 15 -> 25
    deny.toml ............ update (12 lines differ)
      Custom entries detected (1 — would extract to overrides):
        deny: anyhow (wrapper: texting_robots)
    rustfmt.toml ......... ok

  apps/substack-publisher/
    clippy.toml .......... CONFLICT (not managed by guardrail3)
      Run: guardrail3 adopt apps/substack-publisher/clippy.toml
    deny.toml ............ create
    rustfmt.toml ......... create

  Root (packages/):
    clippy.toml .......... create (library profile)
    deny.toml ............ create (library profile)
    rustfmt.toml ......... create

  Project root:
    rust-toolchain.toml .. ok
    release-plz.toml ..... update (3 lines differ)
    cliff.toml ........... ok
    .githooks/pre-commit . update (5 lines differ)

Summary:
  7 files would change (3 create, 4 update)
  4 files up to date
  1 conflict (needs adopt or --force)
  4 custom entries detected (would be preserved via overrides)
```

### TS Dry-Run Output

```
guardrail3 ts generate --dry-run

TypeScript configs:

  Project root:
    .npmrc ................. ok
    tsconfig.base.json ..... update (2 lines differ)
    .jscpd.json ............ ok
    cspell.json ............ update (merge: +0 new settings, preserving 47 custom words)
    .githooks/pre-commit ... update (shared with Rust — 5 lines differ)

  Validated (not generated — checked by validate):
    eslint.config.mjs ...... user-managed (487 lines)
    .stylelintrc.mjs ....... user-managed
    apps/landing/eslint.config.mjs ... user-managed
    apps/landing/tsconfig.json ....... user-managed
    apps/admin/tsconfig.json ......... user-managed

Summary:
  2 files would change (0 create, 2 update)
  2 files up to date
  5 user-managed files (checked by validate, not by generate)
  47 custom words in cspell.json preserved
```

The "Validated" section is informational -- it reminds the user that these files exist and are checked by `validate` even though `generate` doesn't touch them. This prevents the confusion of "I ran generate but my eslint config wasn't updated."

### Color Scheme

For terminal output with color support:
- **create** -- green
- **update** -- yellow
- **ok** -- dim/gray
- **CONFLICT** -- red, bold
- **Custom entries** -- cyan
- Section headers -- bold white

## 5. First-Time vs Subsequent Runs

### First Time (no existing configs)

Detection: No guardrail3-managed files exist on disk (no files with the marker).

Behavior:
1. Generate all files from profile + empty overrides
2. All files classified as "would create"
3. No custom entry detection needed
4. No conflict detection needed

Output (dry-run):
```
guardrail3 generate --dry-run

First-time setup detected. Would generate 11 config files:

  apps/validator-rust/
    clippy.toml .......... create (service profile)
    deny.toml ............ create (service profile)
    rustfmt.toml ......... create
  ...

Run 'guardrail3 generate' to create all files.
```

### First Time with Existing Files (adoption scenario)

Detection: Files exist on disk but none have the guardrail3 marker.

This is the steady-parent scenario: the project already has clippy.toml, deny.toml, etc. but they were written by hand. The user wants guardrail3 to manage them.

Behavior:
1. Generate content from profile + empty overrides
2. Compare against existing files
3. For each file that exists: classify as CONFLICT (unmanaged)
4. Suggest `guardrail3 adopt` for each

Output (dry-run):
```
guardrail3 generate --dry-run

Existing configs detected that are not managed by guardrail3:

  apps/validator-rust/
    clippy.toml .......... CONFLICT (exists, 45 lines, not managed)
      Would extract 8 custom method bans to overrides
    deny.toml ............ CONFLICT (exists, 120 lines, not managed)
      Would extract 1 custom crate ban to overrides

To adopt existing configs:
  guardrail3 adopt apps/validator-rust/clippy.toml apps/validator-rust/deny.toml
  — Extracts custom entries to .guardrail3/overrides/apps/validator-rust/
  — Marks files as guardrail3-managed
  — Regenerates with your customizations preserved

Or to start fresh (DESTRUCTIVE):
  guardrail3 generate --force
```

### Subsequent Runs (normal operation)

Detection: Files exist and have the guardrail3 marker.

This is the common case after initial setup. `generate` regenerates all managed files, applying any override changes.

## 6. The `adopt` Command

New command that bridges the gap between "user has existing configs" and "guardrail3 manages them."

```
guardrail3 adopt <file>...
guardrail3 adopt apps/validator-rust/clippy.toml
guardrail3 adopt --all          # adopt all detected unmanaged config files
```

### Flow

1. Read the existing file
2. Parse it structurally (TOML for clippy/deny, line-by-line for others)
3. Generate the baseline from the current profile
4. Diff: identify entries in the existing file that are NOT in the baseline
5. Write those entries to the appropriate override file:
   - If file is at `apps/{name}/clippy.toml` -> `.guardrail3/overrides/apps/{name}/clippy-methods.toml`
   - If file is at root `clippy.toml` -> `.guardrail3/overrides/clippy-methods.toml`
6. Back up the original to `.guardrail3/backup/`
7. Regenerate the file with marker + baseline + overrides
8. Verify the regenerated file contains all entries from the original (safety check)
9. Report what was done

### Safety Check

After regeneration, `adopt` does a structural comparison: every `{ path = ... }` / `{ name = ... }` entry in the original file must appear in the regenerated file. If any entry is missing, the command fails and restores the backup.

```
adopt: apps/validator-rust/clippy.toml
  Backed up to .guardrail3/backup/apps-validator-rust-clippy.toml.20260318
  Extracted 8 custom method bans to .guardrail3/overrides/apps/validator-rust/clippy-methods.toml
  Regenerated with guardrail3 marker
  Verification: all 47 entries present (39 baseline + 8 custom) -- OK
```

### What `adopt` Does NOT Do

- It does NOT adopt non-TOML files (eslint.config.mjs, .stylelintrc.mjs). These are Validated, not Generated.
- It does NOT modify `guardrail3.toml`. The user must have already configured `[rust.apps.validator-rust]`.
- It does NOT handle structural differences (e.g., different TOML section names). It only handles entry-level differences.

## 7. Subcommand Structure

### Current Structure (keep)

```
guardrail3 generate [--dry-run] [path]     # everything
guardrail3 rs generate [--dry-run] [path]  # Rust only
guardrail3 ts generate [--dry-run] [path]  # TS only
guardrail3 rs diff [path]                  # Rust dry-run (alias)
guardrail3 ts diff [path]                  # TS dry-run (alias)
guardrail3 check [path]                    # CI staleness check
guardrail3 rs check [path]                 # Rust staleness check
```

### Changes

**Merge `diff` into `generate --dry-run`.** The `diff` command is literally `generate --dry-run` with a different name. Having two commands that do the same thing is confusing.

**Decision: Keep `diff` as an alias.** Remove the separate `diff` implementation. Make `rs diff` call `rs generate --dry-run` internally. This way:
- Users who type `diff` get what they expect
- Users who type `generate --dry-run` get what they expect
- One implementation, one output format

**Add `adopt`:**
```
guardrail3 adopt <file>...                 # adopt specific files
guardrail3 adopt --all                     # adopt all detected unmanaged files
```

`adopt` is top-level (not under `rs` or `ts`) because it operates on specific files, not on a language.

**Add `extract`:** This is a lighter version of `adopt` -- it only extracts custom entries to overrides without regenerating the file.

```
guardrail3 extract <file>                  # extract custom entries to overrides
```

Actually -- `extract` is just the first half of `adopt`. Having both is confusing. **Decision: Only `adopt`.** It extracts AND regenerates. If the user wants to see what would be extracted without doing it, they use `adopt --dry-run`.

### Complete Subcommand Tree

```
guardrail3
  generate [--dry-run] [--force] [path]    # generate all config files
  rs
    generate [--dry-run] [--force] [path]  # Rust configs only
    validate [flags] [path]                # validate Rust guardrails
    check [path]                           # CI staleness check
    diff [path]                            # alias for rs generate --dry-run
    init [--profile] [path]                # scaffold guardrail3.toml
    hooks-install [path]                   # install pre-commit hook
    hooks-validate [flags] [path]          # validate hook config
    list-modules                           # list embedded modules
    show-module <name>                     # show module content
  ts
    generate [--dry-run] [--force] [path]  # TS configs only
    validate [flags] [path]                # validate TS guardrails
    diff [path]                            # alias for ts generate --dry-run
    init [path]                            # scaffold TS section
    hooks-install [path]                   # install pre-commit hook
    hooks-validate [flags] [path]          # validate hook config
  adopt [--dry-run] [--all] <file>...      # adopt existing configs
  check [path]                             # CI staleness (all stacks)
  dump-guide                               # generate GUARDRAIL3_GUIDE.md
```

## 8. The `--force` Flag

```
guardrail3 generate --force
```

Behavior: Overwrite ALL files regardless of marker/unmanaged status. Does NOT extract custom entries -- just overwrites.

This is the "I know what I'm doing" escape hatch. Use cases:
- User wants to reset to pure baseline (they'll re-add overrides manually)
- CI pipeline that regenerates everything from scratch

`--force` still:
- Creates backup files in `.guardrail3/backup/`
- Prints what was overwritten
- Warns about custom entries that will be lost

```
guardrail3 generate --force

WARNING: --force will overwrite files that may contain custom content.
Backing up to .guardrail3/backup/

  apps/validator-rust/clippy.toml:
    OVERWRITING — 8 custom method bans will be lost (not in overrides)
    Backed up to .guardrail3/backup/apps-validator-rust-clippy.toml.20260318
  ...

Force-generated 11 files.
Custom entries NOT preserved. To restore: check .guardrail3/backup/
```

## 9. The `check` Command (CI Mode)

`check` is for CI pipelines. It verifies that generated files on disk match what `generate` would produce. Non-zero exit if anything is stale.

### Current Behavior (keep, but enhance output)

```
guardrail3 check

STALE: apps/validator-rust/clippy.toml
STALE: release-plz.toml
MISSING: apps/substack-publisher/deny.toml

3 file(s) need regeneration. Run: guardrail3 generate
```

### Enhancement: Show WHY

For CI debugging, it's helpful to know WHY a file is stale:

```
guardrail3 check

STALE: apps/validator-rust/clippy.toml (3 new baseline bans since last generate)
STALE: release-plz.toml (guardrail3 version bump)
MISSING: apps/substack-publisher/deny.toml (new app in config)

3 file(s) need regeneration. Run: guardrail3 generate
```

The "why" is best-effort -- compare the generated content against disk and report the category of change (new entries, removed entries, setting changes, entirely new file).

## 10. cspell.json Merge Strategy

cspell.json is the one Generated file that needs content merging (preserving the user's custom word list).

### Merge Algorithm

1. Generate the base cspell.json from the cspell module
2. If an existing cspell.json exists on disk:
   a. Parse it as JSON
   b. Extract the `words` array
   c. Merge: `generated_words UNION existing_words`, sorted, deduplicated
   d. Extract any other user-added keys (e.g., `ignorePaths`, `flagWords`)
   e. Merge those into the generated config (user values override generated values for scalar keys; arrays are unioned)
3. Write the merged result

### What NOT to Merge

- `version` -- always use generated
- `language` -- always use generated
- Plugin/dictionary configuration -- always use generated (the user customizes via guardrail3.toml or overrides, not by editing cspell.json directly)

### Dry-Run Output for cspell.json

```
cspell.json ............ update (merge)
  Base config: 12 settings updated
  Custom words: 47 preserved, 0 new from baseline
  User overrides preserved: ignorePaths (3 entries)
```

## 11. Error Handling Matrix

| Scenario | Behavior | Exit Code |
|----------|----------|-----------|
| `guardrail3.toml` not found | Error message, suggest `init` | 1 |
| `guardrail3.toml` parse error | Error message with TOML error detail | 1 |
| Unknown app in config (not discovered) | Warning, skip that app, continue | 0 (with warning) |
| Override file has invalid syntax | Warning per line, skip invalid lines | 0 (with warning) |
| Generated file exists, unmanaged | CONFLICT in dry-run; refuse in generate (suggest adopt) | 1 |
| Generated file exists, managed, content matches | Skip (no-op) | 0 |
| Filesystem write error (permissions, etc.) | Error per file, continue with others | 1 |
| Backup directory creation fails | Error, abort that file | 1 |
| adopt safety check fails (entries missing after regen) | Restore backup, error | 1 |

### Warnings vs Errors

- **Warnings** (yellow, to stderr): Override parse issues, unknown apps, non-critical mismatches. Generate continues.
- **Errors** (red, to stderr): Missing config, write failures, safety check failures. Generate may continue for other files but exits non-zero.
- **Conflicts** (red+bold, to stdout in dry-run): Unmanaged files that would be overwritten. These block `generate` (without `--force`) but `--dry-run` just reports them.

## 12. Interaction with `validate`

`generate` and `validate` are independent but complementary:

- `generate` produces/updates Generated files
- `validate` checks ALL files (Generated + Validated + source code)

After running `generate`, the user should run `validate` to verify the full picture. But `generate` does NOT run `validate` automatically -- they are separate concerns.

**However, `generate --dry-run` SHOULD mention Validated files** (as shown in the TS dry-run output above). This gives the user a complete picture of what guardrail3 cares about, even if `generate` only touches a subset.

## 13. Guardrail3 Marker Format

### TOML Files

```toml
# Generated by guardrail3 v0.5.0 — DO NOT EDIT DIRECTLY
# Profile: service | App: validator-rust
# Customize via .guardrail3/overrides/apps/validator-rust/
# Regenerate: guardrail3 rs generate
```

Line 1 is the detection marker (contains `Generated by guardrail3`). Lines 2-4 are informational.

### JSON Files

```json
{
  "_guardrail3": {
    "generated": true,
    "version": "0.5.0",
    "profile": "service"
  },
  ...
}
```

### Shell Scripts

```bash
#!/usr/bin/env bash
# Generated by guardrail3 — DO NOT EDIT DIRECTLY
# Regenerate: guardrail3 generate
```

### MJS Files (if ever generated)

```javascript
// Generated by guardrail3 — DO NOT EDIT DIRECTLY
// Regenerate: guardrail3 ts generate
```

### Detection Logic

```rust
fn is_guardrail3_managed(content: &str) -> bool {
    content.lines()
        .take(5)  // only check first 5 lines
        .any(|line| line.contains("Generated by guardrail3"))
```

## 14. Migration Path (Adding Markers to Existing Managed Files)

When this feature ships, existing guardrail3 users have managed files WITHOUT markers. The first `generate` run after upgrade would see all its own files as "CONFLICT (unmanaged)."

### Migration strategy:

**Option A: Auto-recognize.** If a file's content matches what `generate` would produce (modulo the marker), treat it as managed and add the marker on the next `generate`. This handles the common case where the user hasn't modified the generated files.

**Option B: One-time flag.** `guardrail3 generate --add-markers` adds markers to all files that match the generated baseline. Files that don't match are flagged as needing `adopt`.

**Decision: Option A (auto-recognize) with fallback to adopt.**

Logic:
1. File exists, no marker
2. Generate the expected content (without marker)
3. If existing content matches expected (ignoring leading comments): auto-add marker, treat as managed
4. If existing content doesn't match: treat as CONFLICT, suggest adopt

This makes the upgrade seamless for users who haven't modified generated files, and guides users who have modified them toward `adopt`.

## 15. Summary: What Changes from Current Implementation

| Current | New |
|---------|-----|
| Global overrides only | Global + per-app overrides |
| No ownership tracking | Marker-based ownership (Generated vs Validated) |
| `warn_if_overwriting` then overwrite anyway | Refuse to overwrite unmanaged files (suggest adopt) |

| Flat file list output | Grouped by location with custom entry details |
| eslint.config.mjs is Generated | eslint.config.mjs is Validated (never generated after init) |
| No `adopt` command | `adopt` bridges existing configs to managed |
| No backup before overwrite | `.guardrail3/backup/` before destructive operations |
| No cspell.json word merge | Merge preserves custom words |
| `check` shows STALE/MISSING | `check` shows STALE/MISSING with reason |
| `generate --dry-run` exists as flag | `generate --dry-run` replaces separate `diff` |

## 16. Open Questions for Future Design

1. **Per-app deny.toml exceptions.** Currently deny.toml is the same for all apps of the same profile. Should apps be able to add per-app crate ban exceptions (e.g., validator-rust allows `anyhow` through a wrapper but substack-publisher doesn't)?

2. **TypeScript per-app generated configs.** Should guardrail3 ever generate per-app tsconfig.json or per-app eslint configs? Current answer: no, these are Validated. But if the pattern proves useful for Rust, it might make sense for TS too.

3. **Override file format evolution.** Currently overrides are raw TOML entry lines. Should they become structured TOML files with sections? E.g.:
   ```toml
   [[methods]]
   path = "reqwest::Client::builder"
   reason = "Use shared clients"
   ```
   This would be more parseable but more verbose. Defer until the current format proves insufficient.

4. **Lock file for generated content.** Should guardrail3 write a `.guardrail3/lock.toml` that records exactly what was generated (hashes, versions, module list)? This would make `check` faster (compare hashes instead of regenerating) and make debugging easier. Worth considering but not required for v1.

5. **Interactive adopt.** Should `adopt` have an interactive mode that shows each custom entry and asks "extract to overrides? / discard? / skip?"? Probably not for v1 -- the non-interactive extract-all is safer and simpler.
