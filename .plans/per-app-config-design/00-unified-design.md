# Unified Design: Per-App Config Architecture for guardrail3

**Date:** 2026-03-18
**Sources:** Plans 01-06 in this directory
**Purpose:** Single synthesis document for the entire per-app config redesign

---

## 1. Executive Summary

guardrail3 is shifting from a "generate whole files from templates and overwrite" model to a layered, non-destructive, per-app-aware system. The core architectural change is threefold: (1) every config file is classified as **Generated** (guardrail3 owns it), **Validated** (user owns it, guardrail3 checks it), or **Shadow-imported** (guardrail3 generates a base file that the user's config imports); (2) overrides move from a flat global directory to a hybrid global + per-app structure with additive and subtractive layers; (3) the `generate` command shifts from "replace entire file" to "ensure guardrail entries are present while preserving user content," using format-appropriate merge strategies (TOML merge via `toml_edit`, JSON key-level merge via `serde_json`, separate import files for JavaScript configs, and line-based merge for INI files). The monorepo story is also being fixed: the single `workspace_root` assumption that causes false positives/negatives in multi-workspace repos is replaced by per-scope validation, and standalone crate apps (not just nested workspaces) are now discovered. The strictness policy is "enforce a floor, not a ceiling" -- user settings that are stricter than the baseline are preserved; settings that are looser are overridden. This design maintains backward compatibility: existing single-app projects and existing override files continue to work unchanged.

## 2. Cross-Cutting Decisions

### File Ownership Model

Three categories, agreed across all plans:

| Category | Meaning | Example |
|---|---|---|
| **Generated** | guardrail3 owns the file entirely. `generate` can overwrite. User customizes via `.guardrail3/overrides/`. | `clippy.toml`, `deny.toml`, `rustfmt.toml`, `rust-toolchain.toml`, `.npmrc`, `tsconfig.base.json` |
| **Shadow-imported** | guardrail3 generates a base/engine file to `.guardrail3/generated/`. The user owns the tool-facing config file, which imports the generated base. | `eslint.config.mjs` (imports engine), `cspell.json` (imports base), `.stylelintrc.mjs` (imports base) |
| **Validated** | User owns the file entirely. guardrail3 never writes it. `validate` checks structural properties. | Per-app `tsconfig.json`, per-app `package.json`, `Cargo.toml [workspace.lints]` |

**Marker system** (Plan 04): Every Generated file starts with `# Generated by guardrail3` (TOML/shell), `// Generated by guardrail3` (JS), or `"_guardrail3": {"generated": true}` (JSON). This enables ownership detection, safe overwrite, and conflict resolution for unmanaged files.

### Override Directory Structure

**Decision: Hybrid global + per-app (Option E from Plan 03).**

```
.guardrail3/
  generated/                              # Shadow bases for JS/JSON configs
    eslint.base.config.mjs
    cspell.base.json
    .stylelintrc.base.mjs
  overrides/                              # Global overrides (all apps)
    clippy-methods.toml
    clippy-types.toml
    clippy-thresholds.toml
    deny-bans.toml
    deny-skip.toml
    deny-feature-bans.toml
    jscpd-overrides.json
    apps/                                 # Per-app overrides
      validator-rust/
        clippy-methods.toml               # ADD entries
        clippy-methods-remove.toml        # REMOVE entries
        clippy-types.toml
        deny-bans.toml
        deny-skip.toml
      substack-publisher/
        clippy-methods-remove.toml
```

App names match `[rust.apps.*]` / `[typescript.apps.*]` keys in `guardrail3.toml`.

### Non-Destructive Editing Strategy Per File Type

| File Type | Strategy | Tool/Crate |
|---|---|---|
| TOML (clippy.toml, deny.toml) | Structured merge at value level | `toml_edit` (preserves comments, formatting) |
| TOML (rustfmt.toml, rust-toolchain.toml) | Structured merge (preserve user additions like extra components) | `toml_edit` |
| JavaScript (eslint.config.mjs, .stylelintrc.mjs) | Separate import file (shadow-base pattern) | N/A -- file separation, not parsing |
| JSON (tsconfig.base.json, .jscpd.json, cspell.json) | Key-level merge | `serde_json` |
| INI (.npmrc) | Line-based key=value merge | Custom line parser |
| Shell (.githooks/pre-commit) | Full replacement | N/A -- guardrail3 owns entirely |

### Per-App vs Global Scoping Per File

| File | Scope | Per-app instances? | Override mechanism |
|---|---|---|---|
| `clippy.toml` | Per-workspace/crate | Yes (one per scope) | Global + per-app additive + per-app subtractive |
| `deny.toml` | Per-workspace/crate | Yes (one per scope) | Global + per-app additive + per-app subtractive |
| `rustfmt.toml` | Per-workspace/crate | Yes (identical content everywhere) | None (no per-app differences) |
| `rust-toolchain.toml` | Repo root only | No | None |
| `release-plz.toml` | Repo root only | No (per-package sections inside) | `[[package]]` sections |
| `cliff.toml` | Repo root only | No | None |
| `eslint.config.mjs` | Root only | No (per-app rules via `files:` scoping) | Shadow-import + user policy |
| `tsconfig.base.json` | Root only | No | Direct merge |
| Per-app `tsconfig.json` | Per-app | Yes (user-managed) | Validated, not generated |
| `.stylelintrc.mjs` | Root only | No | Shadow-import |
| `cspell.json` | Root only | No (per-path overrides inside) | Shadow-import |
| `.npmrc` | Root only | No | Direct merge |
| `.jscpd.json` | Root only | No | JSON merge + override file |
| `.githooks/pre-commit` | Repo root only | No | Full replacement |
| `Cargo.toml [workspace.lints]` | Per-workspace | Yes | Validated, not generated |

### Strictness Policy (Floor Enforcement)

Agreed across Plans 01, 03, and 06:

- **Thresholds** (lower = stricter): keep user value if <= guardrail; override if user is looser.
- **Boolean flags** (strict settings): always enforce the guardrail value (these are binary safety switches).
- **Ban/deny lists**: union merge -- ensure all baseline entries exist, preserve user extras, never remove user-added entries.
- **Rule severities**: enforce floor (`error` > `warn` > `off`; never downgrade).
- **License allow-lists**: ensure baseline licenses present, preserve user extras.

Removals are only via explicit per-app removal files (`*-remove.toml`), never automatic.

### Engine/Policy Split for JS Configs

Agreed in Plan 02, refined in Plans 03 and 06:

- **Engine** = plugins installed, base rules configured, strictness levels set. guardrail3's domain. Goes to `.guardrail3/generated/`.
- **Policy** = module boundaries, import restrictions, per-app overrides, project-specific ignores. User's domain. Lives in the user-owned config file.

guardrail3 validates the engine file for completeness. It validates the user file only for structural properties (imports the engine, required checks present). It does not dictate policy content.

### Override Merge Precedence

Four-layer model, agreed across Plans 01 and 03:

```
Layer 1: Profile baseline (service | library | cli)
Layer 2: Global overrides (.guardrail3/overrides/*.toml)         -- ADDITIVE
Layer 3: Per-app overrides (.guardrail3/overrides/apps/{name}/*) -- ADDITIVE
Layer 4: Per-app removals  (.guardrail3/overrides/apps/{name}/*-remove.toml) -- SUBTRACTIVE
```

Each layer builds on the previous. Removals are processed last so they can remove entries from any prior layer. Deduplication by key (`path` for clippy, `name` for deny) happens after all layers are merged.

## 3. Conflicts Between Plans

### Conflict 1: Stylelint ownership model

**Plan 02** recommends keeping `.stylelintrc.mjs` as a **directly generated** file (guardrail3 overwrites it), because stylelint configs are simple and rarely need project-specific policy.

**Plans 03 and 06** recommend the **shadow-import pattern** for stylelint (same as ESLint), generating to `.guardrail3/generated/.stylelintrc.base.mjs` with a user-owned `.stylelintrc.mjs` that imports it.

**Recommendation:** Use the shadow-import pattern (Plans 03/06). This is consistent with the ESLint approach and prevents the steady-parent problem where custom CSS notation rules and architecture exceptions would be destroyed by regeneration. The cost is minimal (one extra file), and consistency across all JS configs reduces cognitive load.

### Conflict 2: cspell.json ownership model

**Plan 02** recommends the **shadow-import pattern** (`cspell.json` user-owned, imports `.guardrail3/generated/cspell-engine.json`).

**Plan 04** classifies cspell.json as **Generated with merge** -- guardrail3 generates it but preserves the user's `words` array via JSON merge.

**Plan 06** recommends the shadow-import pattern but with direct JSON merge as an alternative.

**Recommendation:** Use the **shadow-import pattern** (Plan 02). cspell natively supports `"import"` for base config extension, making this the cleanest approach. The user's `cspell.json` imports the engine and adds custom `words` -- no merge logic needed, no risk of losing the word list.

### Conflict 3: eslint.config.mjs -- Generated vs Validated vs Shadow-imported

**Plan 02** and **Plan 06** agree on shadow-import (engine file + user-owned config).

**Plan 04** initially classifies it as Validated (guardrail3 never generates after init), then proposes a `mode` config (`"engine"` vs `"standalone"`).

**Recommendation:** Shadow-import with mode config. New projects get `mode = "engine"` (shadow pattern). Existing projects default to `mode = "standalone"` (current behavior) and opt in to engine mode. This is the position Plan 04 converges to.

### Conflict 4: tsconfig.base.json -- Direct merge vs shadow-import

**Plan 06** considers both approaches and decides on **direct JSON merge** (parse existing, set guardrail boolean flags in `compilerOptions`, write back).

**Plan 02** says generate directly (overwrite).

**Recommendation:** Direct JSON merge (Plan 06). tsconfig `extends` has surprising merge behavior for arrays and some settings. Directly setting only the known boolean strict flags in `compilerOptions` while leaving everything else untouched is safer and more predictable.

### Conflict 5: .jscpd.json threshold

**Plan 05 (Gap 4.4):** steady-parent uses threshold 10, guardrail3 generates threshold 0.

**Plan 06:** Proposes enforcing threshold 0 as the floor, overriding if user has higher.

**Recommendation:** This needs a user decision. Threshold 0 means "report all duplicates above minTokens" which is very noisy. Threshold 10 is a practical minimum. The threshold should probably be configurable in `guardrail3.toml` with a sensible default (suggest 5 or 10, not 0).

### Conflict 6: Per-app exemptions location

**Plan 01** proposes a separate `exemptions/` directory under `.guardrail3/apps/{name}/exemptions/`.

**Plan 03** uses `-remove.toml` suffix files in the same `overrides/apps/{name}/` directory.

**Recommendation:** Use the `-remove.toml` suffix approach (Plan 03). A separate `exemptions/` directory adds another level of nesting without benefit. The suffix convention is clear (`clippy-methods.toml` = add, `clippy-methods-remove.toml` = remove) and keeps all per-app customizations in one directory.

## 6. Implementation Phases

### Phase 1: Immediate Fixes (Zero Redesign)

No architectural changes. Fix false positives and missing coverage with minimal code changes.

1. **Add skip flags for eslint/stylelint generation** (Gap 8.4) -- add `eslint` and `stylelint` to `CanonicalConfig`. Trivial config change.
2. **Fix standalone crate discovery** (Gap 1.1) -- `discover_nested_workspaces` fallback for `apps/*/Cargo.toml` with `[package]` but no `[workspace]`.
3. **Fix architecture check path resolution** (Gap 5.6) -- pass `resolve_app_paths` result into architecture check functions.
4. **Fix trivial hook check false positives** (Gaps 7.3, 7.4) -- recognize grep-based tamper detection and jscpd-for-Rust as valid.
5. **Fix .jscpd.json threshold** (Gap 4.4) -- make threshold configurable or use a sensible default.
6. **Add `tools/` and `packages/` to TS discovery** (Gaps 2.1, 2.2) -- expand `discover_ts_apps` to also scan `tools/` and discover packages.

### Phase 2: Core Architectural Changes

The big changes that enable per-app support and multi-workspace validation.

1. **Per-workspace validation loop** (Gap 8.1) -- the single biggest change. Make `rs validate` iterate all discovered workspaces (root, nested, standalone) and run config checks per-workspace-root. This fixes Gaps 5.1, 5.2, 5.3, 5.4, 5.5 as cascade effects.
2. **Per-app override loading** (Gap 8.5) -- extend `load_local_overrides()` to accept an app name and load from `.guardrail3/overrides/apps/{name}/`. Add removal file support (`*-remove.toml`).
3. **Shadow-import pattern for ESLint** (Gap 4.2) -- generate `.guardrail3/generated/eslint.base.config.mjs`. Add `mode = "engine" | "standalone"` config. Scaffold user file on first run.
4. **Shadow-import pattern for stylelint** (Gap 4.3) -- same pattern as ESLint.
5. **Shadow-import pattern for cspell** -- generate `.guardrail3/generated/cspell.base.json`. User's `cspell.json` uses `"import"`.
6. **Marker-based ownership detection** (Plan 04) -- add markers to all generated files. Auto-recognize existing managed files on upgrade.
7. **`adopt` command** (Plan 04) -- extract custom entries from unmanaged files to overrides, mark as managed, regenerate.

### Phase 3: Full Per-App Support

Per-app generation, validation, and overrides working end-to-end.

1. **Generate with per-app context** -- modify `generate_rust_files()` to pass per-app overrides/exemptions through the merge pipeline.
2. **Validate per-scope with per-app expected configs** -- validator computes expected bans per app (profile + global + per-app overrides - removals) and compares.
3. **Per-app tsconfig validation** (Gap 6.3) -- discover per-app tsconfigs, check existence + strict settings.
4. **Per-app eslint validation** (Gap 6.2) -- validate per-app eslint configs for required plugins.
5. **Per-app package.json validation** (Gap 6.6) -- validate devDependencies, scripts, banned deps per app.
6. **TS packages in config model** (Gap 8.3) -- add `packages` to `TypeScriptConfig`.
7. **CLI flag `--app`** -- filter generate/validate/diff to a single app for quick iteration.
8. **Dry-run report redesign** (Plan 04) -- group by location, show override provenance, show custom entry extraction.

### Phase 4: Polish and Advanced Features

Non-destructive editing, advanced merge, and UX improvements.

1. **TOML merge via `toml_edit`** (Plan 06) -- for clippy.toml, deny.toml. Preserves comments and formatting. Enables "ensure guardrail entries, preserve everything else."
2. **JSON merge for tsconfig/jscpd/cspell** (Plan 06) -- key-level merge preserving user content.
3. **INI merge for .npmrc** (Plan 06) -- line-based merge preserving user registry/auth entries.
4. **`check` command enhancement** -- show WHY files are stale, not just STALE/MISSING.
5. **Pre-commit hook monorepo awareness** (Gap 7.2) -- recognize loop-based per-app cargo steps.
6. **`cli` profile** (Plan 01, Open Question 2) -- for CLI tools that don't need service-specific bans.
7. **Unify `diff` into `generate --dry-run`** (Plan 04) -- single implementation, `diff` as alias.
8. **Lock file for generated content** (Plan 04, Q4) -- `.guardrail3/lock.toml` for faster `check`.

## 7. Open Questions

### Requiring User Decision

1. **`.jscpd.json` threshold default.** Plans disagree (0 vs 10). What should the default threshold be? Recommend making it configurable in `guardrail3.toml` with a default of 5 or 10.

2. **Should there be a `cli` profile?** CLI tools like substack-publisher don't need `reqwest::Client::new` banned or `std::process::exit` banned. Currently there's only `service` and `library`. A `cli` profile would be a third baseline. Alternatively, per-app removal overrides can handle this case-by-case.

3. **Should guardrail3 manage `Cargo.toml [workspace.lints]`?** Currently validate-only. For monorepos with multiple workspaces, each needs its own `[workspace.lints]`. Should guardrail3 generate/diff these too, or remain validate-only?

4. **Should packages support per-package overrides?** Plan 01 says no (use `#[allow(...)] // reason:` per-function instead). But what if a math library needs `float_cmp` exempted project-wide?

5. **ESLint engine file: flat array or named exports?** Flat array (`export default [...]`) is simpler. Named exports (`export { baseRules, tsStrictRules, reactRules }`) let users selectively import. More flexible but more complex.

### Technical Decisions Not Yet Resolved

6. **JSONC support for tsconfig.** If the user's `tsconfig.base.json` contains `//` comments, `serde_json` will fail. Options: strip comments before parsing (lossy), use a JSONC-aware parser, or fall back to shadow-import pattern for JSONC files.

7. **How do pre-commit hooks work with multiple workspaces?** The hook runs from repo root. It needs to run `cargo clippy` / `cargo deny` / `cargo test` for EACH workspace independently. The current hook template assumes one workspace. Plan 01 flags this but doesn't propose a solution.

8. **Should `check` and `validate` merge for config files?** With non-destructive editing, `check` becomes "verify guardrail entries are present" which overlaps heavily with `validate`. Plan 06 raises the question but defers.

9. **How should `guardrail3 init` handle monorepos?** Currently creates a single `guardrail3.toml` entry. For monorepos, it should discover all workspaces and standalone crates and generate all `[rust.apps.*]` entries. Plan 01 flags this.

10. **Should `.guardrail3/generated/` files be committed?** Plan 03 says yes (CI can run without `guardrail3 generate` first; `guardrail3 check` verifies staleness). This matches the current behavior where generated configs are committed.

11. **How does the `mode = "engine"` migration work for existing projects?** Plan 02 proposes: backup existing eslint config, generate engine file, scaffold new user config, print instructions to merge. Plan 04 proposes: auto-recognize existing managed files. These need to be reconciled into a single migration path.

12. **Should `validate` detect user overrides that weaken engine rules?** Plan 02 proposes T-POLICY-WEAKEN (Info severity) when the user's eslint.config.mjs overrides an engine rule to `off` or `warn`. This is audit-only, not blocking.
