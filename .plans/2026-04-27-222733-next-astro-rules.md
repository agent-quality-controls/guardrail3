# Goal

Record the next obvious Astro hardening rules before starting the rule ID migration.

# Rules To Build Next

1. Suppression detection for strict Astro apps.
   - Detect `eslint-disable`, `eslint-disable-next-line`, and `eslint-disable-line` that suppress required delegated rules.
   - Scope: Astro source, TS/TSX routes, MDX content, and MDX component-map files.
   - Owner: Astro setup/content/MDX checks, depending on which delegated rules are suppressed.
   - Reason: G3TS delegates heavily to ESLint; one suppression comment must not bypass the guardrail model silently.
   - Implementation constraint: use ESLint parser/API facts if available. Do not hand-roll comment regex unless there is no parser-backed option.

2. Whole-app validation script contract.
   - Require a fail-closed script such as `check` or `validate` that runs lint, package lint, typecheck, and build.
   - Owner: Astro setup config checks.
   - Reason: package scripts must make the complete validation chain obvious and runnable by agents and CI.
   - Implementation constraint: use the existing package script command parser, not substring checks.

3. Astro content collection schema depth.
   - Require configured public collections to define schemas with required fields from `guardrail3-ts.toml`.
   - Owner: Astro content checks.
   - Reason: Astro content collections exist, but an empty or shallow schema still lets agents move weak content contracts into components.
   - Implementation constraint: do not hardcode one blog schema. Required fields must come from explicit policy config.

4. Published-content draft/future policy.
   - Public static builds must not publish draft or future-dated content unless policy explicitly allows it.
   - Owner: Astro content checks or a delegated app content validation script enforced by Astro content checks.
   - Reason: content visibility is part of the static Astro content pipeline.
   - Non-scope: link checking.

5. Image wrapper schema policy.
   - Approved MDX image wrappers must validate image props such as `src`, `alt`, and project-required image policy fields.
   - Owner: Astro MDX checks plus `g3ts-eslint-plugin-astro-pipeline` if source-level wrapper enforcement is needed.
   - Reason: raw image syntax is banned, but weak wrapper schemas can still accept unsafe image data.

# Priority

Build suppression detection first. It protects every delegated ESLint rule already added.
