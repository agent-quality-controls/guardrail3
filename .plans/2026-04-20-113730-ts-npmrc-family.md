## Goal

Create `packages/ts/npmrc` as the root `.npmrc` validation family.

End state for wave 1:

- discovers the package-manager-root `.npmrc`
- parses it line-by-line
- inventories duplicate keys
- validates a narrow, durable baseline
- preserves strict separation from `ts/package`

## Local Intent

Local plans already treat `ts/npmrc` as its own family. It is meant to own:

- root `.npmrc` existence
- duplicate-key detection
- required baseline settings
- weaker-than-baseline values
- extra-setting inventory

The old implementation is mixed into the broad config blob and hardcodes a 13-key baseline.

## External 2026 Delta

The old `.npmrc` baseline looks too broad against current docs:

- pnpm now splits configuration more clearly between `.npmrc` and `pnpm-workspace.yaml`
- pnpm treats some old baseline knobs as broader workspace/hoisting/settings concerns
- npm v11.2+ warns on unknown `.npmrc` keys
- package-manager enforcement is now more explicitly split across:
  - `package.json#packageManager`
  - npm `devEngines.packageManager`
  - pnpm `engines.pnpm`

That means the legacy all-in-one `.npmrc` contract should not be migrated unchanged.

## Approach

1. Treat `.npmrc` as a root-only config family.
2. Parse line-by-line so comments, auth lines, and raw formatting survive inspection.
3. Split rule concerns into:
   - exists
   - duplicate keys
   - required stable settings
   - relaxed values
   - extra-setting inventory
4. Keep `package.json` package-manager and engine policy out of this family.

## Wave 1 Rule Cut

Wave 1 should include only narrow, stable `.npmrc` policy:

- root `.npmrc` exists
- root `.npmrc` parses line-by-line
- duplicate keys are flagged
- stable root settings are checked where current docs still support the surface clearly

Wave 1 should not yet carry forward every old key from the legacy baseline.

Specifically, these should stay out of wave 1 until ownership and current support are re-verified:

- `save-workspace-protocol`
- `package-manager-strict-version`
- `strict-dep-builds`
- `verify-deps-before-run`
- `public-hoist-pattern`
- `shamefully-hoist`
- `save-prefix`

## Key Decisions

- Keep `.npmrc` narrow.
  - Reason: current npm and pnpm docs do not justify treating it as a giant package-policy surface anymore.

- Keep it root-only.
  - Reason: current local notes and pnpm behavior point to one package-manager-root `.npmrc`, not per-package enforcement.

- Preserve auth/registry lines as first-class parseable input.
  - Reason: these lines are valid and must not be mistaken for policy errors.

## Latest Ideas, Not Final Decisions

- Some of the legacy `.npmrc` baseline may belong in `pnpm-workspace.yaml` instead of `.npmrc`.
- `ts/npmrc` may end up intentionally much smaller than the old checker.
- Package-manager enforcement may shift further toward `package.json` and away from `.npmrc`.

## Risks / Open Questions

- Which old pnpm settings are still current, and which now belong to workspace settings instead.
- Whether `devEngines.packageManager` should become part of `ts/package` wave 2.
- Whether a future `pnpm-workspace` family is needed rather than overloading `.npmrc`.
