# Parser package follow-up: cargo-config-toml and guardrail config split

This note replaces the earlier idea of one shared `guardrail3-toml` parser.

The config direction is now:
- Rust config moves to `guardrail3-rs.toml`
- TypeScript config moves to `guardrail3-ts.toml`
- parser packages should mirror that split instead of preserving one shared schema

Important:
- this is a target design note, not completed implementation
- the current app code still reads `guardrail3.toml`
- Rust is the active roadmap; TypeScript config work is documented only enough to keep the split explicit

## cargo-config-toml

Parser for `.cargo/config.toml` and `.cargo/config`.

Status:
- implemented

Used by guardrail3 today:
- clippy family
- current rule-owned need: `[env].CLIPPY_CONF_DIR`

Schema policy:
- type known Cargo sections/fields we can state confidently
- preserve unknown keys in `extra`
- use `toml::Value` only where Cargo itself allows open or multi-shape values

## guardrail3-rs-toml

Parser for `guardrail3-rs.toml`.

This should be a Rust-only policy file. It should not carry TypeScript config branches or TS-driven aliases.

### Why split

Current shared `guardrail3.toml` shape mixes:
- Rust policy and family toggles
- TypeScript config and generation settings
- hooks and shared top-level profile concepts

That creates fake coupling:
- Rust and TS do not appear to share meaningful rule semantics
- the current `type` alias exists largely because the schema was trying to span both worlds
- parser packages should model real ownership boundaries, not historical config convenience

### Rust-only field structure proposal

The Rust-only file should flatten away the old outer `[rust]` wrapper, remove the `type` alias, and configure exactly one workspace per file.

Proposed shape:

```toml
version = "1"
excluded_paths = ["legacy", "tests/fixtures"]
profile = "service"
allowed_deps = ["serde", "toml"]

[checks]
topology = true
arch = true
fmt = true
toolchain = true
clippy = true
deny = true
cargo = true
code = true
hexarch = true
deps = true
garde = true
test = true
release = true
hooks_shared = true
hooks_rs = true

[[waivers]]
rule = "RS-CARGO-12"
file = "Cargo.toml"
selector = "clippy:redundant_pub_crate"
reason = "Documented temporary exception"
```

### Proposed typed model

```rust
pub struct Guardrail3RsToml {
    pub version: Option<String>,
    pub profile: Option<RustProfile>,
    #[serde(default)]
    pub excluded_paths: Vec<String>,
    #[serde(default)]
    pub allowed_deps: Vec<String>,
    pub checks: Option<RustChecksConfig>,
    #[serde(default)]
    pub waivers: Vec<WaiverConfig>,
    #[serde(flatten)]
    pub extra: BTreeMap<String, Value>,
}

pub enum RustProfile {
    Service,
    Library,
}

pub struct RustChecksConfig {
    pub topology: Option<bool>,
    pub arch: Option<bool>,
    pub fmt: Option<bool>,
    pub toolchain: Option<bool>,
    pub clippy: Option<bool>,
    pub deny: Option<bool>,
    pub cargo: Option<bool>,
    pub code: Option<bool>,
    pub hexarch: Option<bool>,
    pub deps: Option<bool>,
    pub garde: Option<bool>,
    pub test: Option<bool>,
    pub release: Option<bool>,
    pub hooks_shared: Option<bool>,
    pub hooks_rs: Option<bool>,
    #[serde(flatten)]
    pub extra: BTreeMap<String, Value>,
}

pub struct WaiverConfig {
    pub rule: String,
    pub file: String,
    pub selector: String,
    pub reason: String,
    #[serde(flatten)]
    pub extra: BTreeMap<String, Value>,
}
```

### Deliberate changes from the current shared schema

- remove outer `[rust]`
  The file is Rust-only, so repeating `rust.*` is noise.
- remove `type`
  Rust should use one selector name: `profile`.
- replace app/package subtrees with one workspace profile
  The file configures exactly one workspace, so the workspace itself has one profile.
- drop `workspaces`
  Current Rust code appears to rely on Cargo discovery, not on a configured workspaces list.
- keep `excluded_paths`
  Rust legality/runtime still consume it.
- keep `checks`, `allowed_deps`, `waivers`
  These remain plausible workspace-local policy inputs.

### Current Rust-owned fields that justify this shape

Observed live consumers today:
- family enable/disable toggles
- dependency allowlist policy via `allowed_deps`
- Rust-only `excluded_paths`
- shared waiver inventory used by Rust families

### Fields intentionally not carried into the Rust-only file

- `typescript`
- TS app metadata
- TS canonical generation settings
- TS checks
- hooks as a general cross-language config branch

Rust hook-family toggles stay under Rust `checks` because those are Rust validation families.

### Waiver format

The current generic `escape_hatches` shape in the legacy config is too vague. Different families reuse the same tuple fields for different meanings, which makes the abstraction dishonest.

The candidate replacement is a simpler generic waiver list:

```toml
[[waivers]]
rule = "RS-CARGO-12"
file = "Cargo.toml"
selector = "clippy:redundant_pub_crate"
reason = "workspace lint policy requires this temporary allow"

[[waivers]]
rule = "RS-FMT-07"
file = "rustfmt.toml"
selector = "ignore"
reason = "generated ffi bindings are unstable under rustfmt"

[[waivers]]
rule = "RS-GARDE-09"
file = "src/adapters/db/user_queries.rs"
selector = "sqlx::query_as@L42"
reason = "query maps into a legacy row type that cannot derive garde yet"
```

Shape:

- `rule`
  - exact rule id, e.g. `RS-CARGO-12`
- `file`
  - exact file path the waiver applies to
- `selector`
  - exact rule-defined selector string for the waived subject inside that file
- `reason`
  - human explanation

Non-goals:

- no family-wide waivers
- no globs
- no regex selectors
- no generic `kind` field
- no expiration or fingerprint machinery

### Waiver count visibility

Even when a waiver is valid, families should continue to emit inventory/warning results for waiver count.

Reason:

- waivers are not invisible success
- a cleanly justified waiver is still policy debt
- count/inventory output keeps the waiver surface visible in reports

Expected family behavior:

- no matching waiver:
  - emit the normal failing result
- matching waiver with weak reason:
  - emit a failing result about weak justification
- matching waiver with valid reason:
  - emit a non-failing tracked/inventory result
- whenever one or more waivers are present for the family/root/file:
  - also emit a count/inventory result summarizing how many waivers were used

### Selector ownership design

Waiver matching should be rule-driven, not config-driven.

That means:

- the parser only reads `[[waivers]]` into typed data
- runtime indexes waivers by `(rule, file, selector)`
- each waivable rule defines its own selector format
- the rule computes the selector for each violation candidate it finds
- matching is exact string equality on `(rule, file, selector)`

Examples:

- `RS-CARGO-12`
  - selector format: `{family}:{lint}`
  - example: `clippy:redundant_pub_crate`
- `RS-FMT-07`
  - selector format: fixed string
  - example: `ignore`
- `RS-GARDE-09`
  - selector format: `{macro_name}@L{line}`
  - example: `sqlx::query_as@L42`
- `RS-HEXARCH-16`
  - selector format: `{patch_key}@{resolved_rel_dir}`
  - example: `foo@crates/bar`

Implication:

- a rule that cannot define a clean exact selector should not support waivers
- selector semantics belong to the rule implementation, not the parser schema
- the generic waiver envelope stays simple while rule matching remains precise

### Profile rule

The new Rust-only file should use exactly one explicit workspace profile.

That means:
- the workspace root must set `profile = "service"` or `profile = "library"`
- there is no root-level fallback like `[profile].name`
- there are no app/package-local profile overrides in this file
- parser and validation should fail closed when an adopted workspace omits `profile`

This is intentionally stricter than the current shared schema. The goal is to remove inherited ambiguity and make policy ownership obvious at the workspace boundary.

## Workspace-scoped guardrail research draft

This section is research only. It is a candidate direction for discussion, not a decided architecture.

### Working direction under discussion

- no global Rust families or repo-global Rust governance
- opt-in per Cargo workspace
- all Rust families become workspace-scoped
- hooks route staged files to the owning adopted workspace instead of running one repo-global Rust pass

### Candidate workspace contract

Use two separate concepts:

- `candidate workspace`
  - nearest ancestor directory with a `Cargo.toml` containing `[workspace]`
- `adopted workspace`
  - candidate workspace whose `Cargo.toml` opts into guardrail3

The current leading idea is to put the adoption marker in Cargo metadata instead of inventing a second top-level config surface.

Candidate shape:

```toml
[workspace.metadata.guardrail3]
enabled = true
```

Under this model:

- `Cargo.toml` defines the workspace boundary and adoption marker
- sibling `guardrail3-rs.toml` defines policy for that workspace
- if Cargo opts in but `guardrail3-rs.toml` is missing, that should be a hard failure for the workspace

This keeps structural ownership in Cargo and policy ownership in the guardrail config file.

### Hook routing concept

If Rust validation is workspace-scoped, the hook can route staged files by ownership:

1. collect staged paths from Git
2. walk upward from each staged path to the nearest workspace root
3. if no workspace root exists, skip Rust guardrail checks for that path
4. if a workspace root exists but does not opt in, skip Rust guardrail checks for that path
5. if the workspace root opts in, run guardrail3 once for that workspace with staged files under that root as validation scope

This model avoids a repo-global Rust pass while still validating the correct workspace for each staged change.

### Scope consequences this direction is trying to achieve

The intended effect is:

- random monorepo Rust outside adopted workspaces does not get linted by default
- every adopted workspace owns its own Rust policy boundary
- there is no repo-global fallback where one root config implicitly governs unrelated Rust surfaces

Open consequence questions still under discussion:

- what, if anything, should happen to Rust files outside any adopted workspace
- whether hooks remain purely workspace-owned or need any separate shared layer later
- how to handle staged files that are not under any Cargo workspace

### Family model under this direction

If this direction holds, then every Rust family becomes workspace-scoped.

That means:

- no global Rust topology family
- no repo-global code/test/fmt/arch passes
- no repo-global Rust governance file
- every family result belongs to exactly one adopted workspace

This is intentionally a larger architectural shift than “just add a new parser package.”

### Extracted family package shape

The current direction under discussion is to extract family checks into two package types per family:

- `g3-<family>-content-checks`
- `g3-<family>-filetree-checks`

Intended split:

- `content-checks`
  - pure semantic checks over typed parsed inputs
  - depend on `*-toml` parser packages and other typed content models
- `filetree-checks`
  - structural placement, presence, ownership, and relationship checks
  - depend on workspace/filetree facts, not raw text parsing

This would keep family check crates honest about what they consume:

- parsed config semantics come from parser packages
- workspace structure semantics come from orchestrator-built facts
- no regex/string matching on config semantics inside family rules

### Parser dependency direction

Per-family extracted check packages should depend on filename-specific parser packages where configuration semantics are involved.

Examples:

- `rustfmt-toml`
- `cargo-config-toml`
- `clippy-toml`
- `deny-toml`
- `nextest-toml`
- `mutants-toml`
- `rust-toolchain-toml`

Family orchestrators should own:

- discovery
- file loading
- parser invocation
- normalization into minimal rule inputs

The extracted check packages should consume typed inputs, not file IO and not ad hoc parsing.

### Migration sketch for this direction

If this direction is chosen later, the likely order is:

1. freeze the workspace adoption contract in `Cargo.toml`
2. freeze the `guardrail3-rs.toml` workspace-local schema
3. remove repo-global assumptions from Rust family selection, routing, and runtime
4. convert all remaining global families to workspace-local semantics
5. teach the hook to map staged files to owning adopted workspaces
6. extract family rules into `content-checks` and `filetree-checks` packages

## guardrail3-ts-toml

Target file:
- `guardrail3-ts.toml`

Status:
- deferred

Reason:
- TypeScript is not the active roadmap right now
- we should keep the split explicit, but not spend implementation time inventing a TS schema before there is an active TS workstream

Current design rule:
- do not preserve one shared `guardrail3-toml` parser just to avoid making the split
- when TS work resumes, define its schema independently from the Rust file

## Build order

1. cargo-config-toml
2. document the guardrail config split
3. define and freeze `guardrail3-rs.toml`
4. build `guardrail3-rs-toml`
5. migrate Rust consumers off `guardrail3.toml`
6. change Rust discovery/runtime so only roots with `guardrail3-rs.toml` opt in
7. only then decide whether `guardrail3-ts.toml` needs a parser package at all
