# Rule Splitting

Rules should check one thing. These rules currently check multiple things and need to be split.

## RS-ARCH-02: lib.rs facade-only
Currently checks two things:
1. Implementation logic in lib.rs (body_items: functions, impl blocks, etc.)
2. Broad re-exports in lib.rs (glob/wildcard `pub use`)

These should be separate rules.

## RS-ARCH-03: mod.rs required (borderline)
Checks one concept (does mod.rs exist?) but has two distinct violation types:
1. foo.rs-alongside-foo/ convention (wrong placement)
2. Missing mod.rs entirely (absent)

These are arguably different violations. Consider splitting.

## RS-ARCH-08: feature-gated exports
6 distinct checks in one rule:
1. Ungated pub exports in lib.rs
2. Exports gated directly on `all` instead of named features
3. Missing `all` feature in Cargo.toml
4. Empty `all` feature
5. Missing `default` feature
6. `default` doesn't include `all`

Each should be its own rule. A crate with no feature gates triggers 1, 3, and 5 simultaneously — cascade noise.

## RS-TOPOLOGY-01, 03, 04: Possible consolidation
These three rules check the same underlying problem (zone overlap) from different angles:
- 01: per-crate symptom (ambiguous classification)
- 03: per-crate symptom (dual ownership)
- 04: zone-level cause (zones physically nest)

The fix is always "restructure so zones don't overlap." Consider consolidating.

## Libarch orphans — research needed
Multiple rules only fire for `profile_name == "library"`, which was from the now-removed libarch family:
- RS-DEPS-08: Library Allowlist Present
- RS-CLIPPY-14: Library Global State type bans
- RS-CARGO-15: Library rust-version requirement
- RS-CARGO-01: Library-specific rust lints

Need to determine: should these rules still exist? Should they apply to all crates? Is "library" profile meaningful without libarch?

## RS-DEPS-08: Library Allowlist Present — research needed
Only fires for crates with `profile_name == "library"`. The library profile was from the now-removed libarch family. Need to determine:
- Should this rule still exist?
- Should allowlists apply to all crates, not just "library" profile?
- Is the "library" profile concept still meaningful without libarch?

## RS-TOOLCHAIN-02: Channel and Components — split
Three checks in one rule:
1. TOML parse validity
2. Channel policy (stable/pinned required)
3. Components policy (clippy, rustfmt required)

## RS-TOPOLOGY-12: Declared Workspace Members Only — split
Two distinct checks in one rule:
1. Undeclared workspace member (crate exists but not in members list)
2. Extra workspace member (in members list but no matching crate)

These are opposite violations and should be separate rules.

## RS-TOPOLOGY-09, 10: Possible consolidation
- 09: top-level root must declare `[workspace]`
- 10: top-level root has `[package]` but no `[workspace]`
10 is a subset of 09. Both fire on the same underlying problem. Consider consolidating.

## RS-TOPOLOGY-06: Owner Family Enablement Coherence — probably remove
- References "libarch" which no longer exists as a family
- The rule logic is wrong: having a crate under `apps/` should not force enabling hexarch checks
- Candidate for removal

## RS-ARCH-05: no boundary crossing (edge case)
When `boundary_rel` is empty, the label becomes "project root" and the fix instruction "Depend on project root's facade crate" doesn't make practical sense. Needs special handling for the root boundary case.

## RS-ARCH-04: mod.rs facade-only
Same pattern as ARCH-02 — checks both body items and broad re-exports in mod.rs. Should also be split.
