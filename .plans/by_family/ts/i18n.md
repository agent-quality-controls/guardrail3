# TS-I18N

Status: current family contract, legacy-grouped implementation, but currently overloaded into one bucket rule.

Implementation roots:

- `apps/guardrail3/crates/app/ts/validate/i18n_check.rs`

Current source of truth:

- this file for family planning/status
- `.plans/todo/checks/ts/i18n.md` as the detailed family ledger until the cutover is complete

Current state:

- i18n checks exist as a distinct validator file, but not as a dedicated family surface
- compared with Rust family design standards, this family is still too bundled and needs a sharper ownership boundary against `TS-CONTENT` and `TS-SEO`

Rule inventory:

- `T-TOOL-12` — i18n enablement and locale completeness.
  - What it should do: detect an i18n library and then enforce the current completeness contract:
    - an i18n message directory exists
    - at least two locale JSON files exist
    - locale files share the same top-level keys
  - What it is for: this turns i18n from “library installed” into an actual translation-surface contract.

Current code mapping:

- `apps/guardrail3/crates/app/ts/validate/i18n_check.rs`
  - currently overloads all live i18n checks under `T-TOOL-12`
  - detects libraries in `package.json`
  - finds a message directory
  - checks that multiple locale files exist
  - compares top-level locale-key parity

Current doc/code reconciliation notes:

- the old ledger is broader than the current runtime; it talks about locale routing and translation-surface correctness that are not yet implemented here
- the current code overloads one ID (`T-TOOL-12`) across multiple distinct i18n checks, so this family is a good candidate for a future split into family-native IDs
- this family should remain a capability family, not an owner family; content/public-web root ownership should come from `TS-ARCH` or `TS-CONTENT`
- the live rule currently mixes:
  - enablement/tool detection
  - directory presence
  - locale-count requirements
  - locale-key parity
  Those should eventually become separate family-native rules
- the family still lacks explicit fail-closed wording for malformed or unreadable locale/config inputs
