# Websmasher MVP Architecture Decisions So Far

## Goal

- Capture the architecture decisions that are already settled
- Reduce drift across scattered handoff and decision notes
- Make the remaining design work explicit

## Product shape

- Websmasher is a multi-service product:
  - `web` - Next.js dashboard and BFF
  - `landing` - marketing site
  - `backend` - Rust/Axum control plane and business logic
  - `crawler` - Python scrape and extraction service

## Frontend and service shape

- `web` is the auth edge for browser traffic
- `web` acts as a BFF between UI and `backend`
- `web` reads app state from Postgres through repositories
- `web` does not own core business writes
- `backend` owns business workflows, orchestration, run state, and validation
- `crawler` owns scrape execution and artifact production

## Auth

### Provider

- Auth provider: PropelAuth

### Org model

- Websmasher is org-scoped internally from day 1
- Every user gets one private default org for MVP
- MVP supports exactly one org per user
- No org switcher in Websmasher MVP
- No org management UI in Websmasher MVP
- PropelAuth-hosted pages are the fallback for org management if needed before custom UI exists

### Local identity model

- App uses local identity projection from day 1:
  - `users`
  - `orgs`
  - `memberships`
- App-owned IDs are the business IDs
- `org_id` in product data is app-owned
- Provider IDs such as `propelauth_org_id` are mapping fields, not primary business IDs

### Provisioning boundary

- `web` is the provisioning/auth edge
- `backend` does not know PropelAuth internals
- Default-org provisioning runs from `web`
- Canonical trigger: first authenticated server-side app entry
- Provisioning must be idempotent

### Roles

- MVP roles:
  - `owner`
  - `member`
- Every provisioned MVP user starts as `owner`
- No in-app role management UI in MVP

### Web to backend trust contract

- `web` authenticates the browser session with PropelAuth
- `web` mints a short-lived internal JWT for calls to `backend`
- `backend` verifies that internal JWT
- `backend` does not verify PropelAuth browser session tokens directly

Minimum identity context:

```json
{
  "actor_type": "user",
  "user_id": "<local_user_id>",
  "org_id": "<local_org_id>",
  "org_role": "owner"
}
```

Rules:
- request-scoped
- server-generated by `web`
- not sourced from browser input
- `request_id` and tracing metadata are not auth claims

## Error model

- `backend` and `crawler` expose HTTP errors as Problem Details
- Websmasher extends Problem Details with:
  - `code`
  - `retryable`
  - `field_errors`
- Expected failures are explicit structured outcomes
- Unexpected failures are generic at the boundary and detailed in logs
- Stable low-cardinality `code` values are the cross-service machine-readable contract

## Crawl execution and status ownership

- `backend` owns run state and phase transitions
- `crawler` does not own job state of record
- `crawler` reports progress and completion back to `backend` via webhook callbacks
- `crawler` does not expose a job status endpoint
- `web` reads run status from Postgres through repositories
- UI status is read through `web`

Crawler callback events already planned:
- `discovery_complete`
- `page_complete`
- `scrape_complete`
- `extraction_complete`

## What is intentionally deferred

- Multi-org support in product UX
- Invite flow in app UI
- Team/org management UI
- Rich RBAC
- Custom realtime transport beyond simple UI polling if polling is sufficient
- Detailed local identity table design
- Detailed provisioning sequence in `web`
- Forms conventions
- Testing strategy

## Remaining undecided areas

### 1. Forms

Need to decide:
- React Hook Form + Zod or alternative
- where schemas live
- how validation errors return to UI
- when to use server actions vs client mutations

### 2. Testing strategy

Need to decide:
- what gets unit tested
- what gets integration tested
- what gets component tested
- what minimal e2e path exists

### 3. Detailed local identity schema

Need to decide later:
- exact `users`, `orgs`, `memberships` columns
- exact provider mapping fields
- uniqueness rules and idempotency constraints

### 4. Detailed provisioning flow

Need to decide later:
- exact server entry point in `web`
- exact ordering of local upserts vs PropelAuth calls
- exact failure handling path
