## Goal

Build `packages/ts/jscpd` wave 1 and wire it into `g3ts`.

End state:

- dedicated parser package for `.jscpd.json`
- TS family packages:
  - `g3ts-jscpd-types`
  - `g3ts-jscpd-ingestion`
  - `g3ts-jscpd-config-checks`
- `g3ts validate --family jscpd` works on explicit target roots
- the live `websmasher` repo root validates cleanly under the new family

## Approach

1. Add `packages/parsers/jscpd-json-parser`.
   - parse JSON
   - expose typed `JscpdDocument { raw, typed }`
   - keep helper accessors parser-owned
2. Build `packages/ts/jscpd/g3ts-jscpd-types`.
   - expose a narrowed family snapshot, not parser documents
3. Build `g3ts-jscpd-ingestion`.
   - select root `.jscpd.json` for the explicit target root
   - parse once
   - convert into family-owned snapshot
4. Build `g3ts-jscpd-config-checks`.
   - pure rules only
   - wave 1:
     - config exists
     - config parses
     - `threshold` is `0`
     - `absolute` is `true`
     - required ignore patterns exist
     - `format` exists and includes `typescript`
5. Wire `SupportedFamily::Jscpd` into `apps/guardrail3-ts`.
6. Verify on the `websmasher` repo root.

## Key Decisions

- Keep wave 1 root-only for the explicit target root.
  - Why: `.jscpd.json` is a config-root surface, and the live target already has one at repo root.

- Keep `format` ownership narrow.
  - Why: this family should prove that TS duplication detection is configured for TypeScript, not claim unrelated content-site policy from the legacy file.

- Do not carry legacy content checks from `jscpd_check.rs`.
  - Why: those belong to `ts/content`, not `ts/jscpd`.
