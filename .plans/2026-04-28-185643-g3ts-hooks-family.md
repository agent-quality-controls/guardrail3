# Goal

Plan the G3TS hooks family by copying the working G3RS hook architecture where it is actually good:

- shared hook structure and shell safety rules stay language-agnostic
- TypeScript-specific hook checks only prove TypeScript validators run from the effective pre-commit hook
- no raw substring matching for shell scripts
- no Astro-specific hook package
- no duplicate implementation of validators already owned by Astro, Style, or package families

# G3RS Reference

The active G3RS hook implementation is under:

- `packages/rs/hooks/g3rs-hooks-types`
- `packages/rs/hooks/g3rs-hooks-ingestion`
- `packages/rs/hooks/g3rs-hooks-file-tree-checks`
- `packages/rs/hooks/g3rs-hooks-source-checks`
- `packages/rs/hooks/g3rs-hooks-config-checks`

G3RS plan references:

- `.plans/todo/checks/hooks/shared.md`
- `.plans/todo/checks/hooks/rs.md`

Important implementation facts:

- G3RS merged language-agnostic hook rules and Rust-specific hook rules into one `g3rs-hooks` package family.
- File tree checks own hook existence, `core.hooksPath`, executable bit, modular script inventory, override inventory, and hook-system shadowing risks.
- Source checks own shell script semantics and Rust hook command presence.
- Config checks own installed tool availability.
- Ingestion owns hook surface discovery and calls `hook-shell-parser::parse_script`.
- Rules receive typed hook facts, not filesystem handles.
- Shell commands are checked through parser facts. Comments, `echo`, and inert text do not count.

# Package Structure

Create:

- `packages/ts/hooks/g3ts-hooks-types`
- `packages/ts/hooks/g3ts-hooks-ingestion`
- `packages/ts/hooks/g3ts-hooks-file-tree-checks`
- `packages/ts/hooks/g3ts-hooks-source-checks`
- `packages/ts/hooks/g3ts-hooks-config-checks`

Do not create:

- `g3ts-astro-hooks`
- `g3ts-content-hooks`
- one hook package per TS framework

Reason:

- hook structure is repository-local and language-agnostic
- hook command semantics are TypeScript ecosystem-specific, not Astro-specific
- Astro already owns the app-level `validate` script contract
- hooks only need to prove the effective hook invokes that app validation command where TS apps exist

# Better Shared Architecture For G3RS And G3TS

The current G3RS hook family is useful, but it is still too self-contained:

- Rust hook rules know which Rust tools should run.
- Rust hook rules know which Rust config files should trigger validation.
- Shared shell rules live under `packages/rs/hooks`, even though they are not Rust-specific.

The better architecture is:

1. Each validation family declares its hook requirements in its owned policy/check contract.
2. A shared hook family discovers and parses the effective hook once.
3. The language hook family aggregates declared requirements for that language.
4. Hook rules verify the aggregated requirements against parsed executable command facts.

This should be applied to both G3RS and G3TS.

## Family Hook Contract

Each family that needs hook enforcement must have a dedicated hook-contract package.

Package pattern:

- `g3rs-<family>-types`
- `g3rs-<family>-ingestion`
- `g3rs-<family>-config-checks`
- `g3rs-<family>-source-checks` if needed
- `g3rs-<family>-file-tree-checks` if needed
- `g3rs-<family>-hook-contract`

TypeScript equivalent:

- `g3ts-<family>-types`
- `g3ts-<family>-ingestion`
- `g3ts-<family>-config-checks`
- `g3ts-<family>-source-checks` if needed
- `g3ts-<family>-file-tree-checks` if needed
- `g3ts-<family>-hook-contract`

The hook-contract package is the only package in a family that defines what the family requires from hooks.

Example Rust folders:

- `packages/rs/fmt/g3rs-fmt-hook-contract`
- `packages/rs/clippy/g3rs-clippy-hook-contract`
- `packages/rs/deny/g3rs-deny-hook-contract`
- `packages/rs/cargo/g3rs-cargo-hook-contract`
- `packages/rs/test/g3rs-test-hook-contract`
- `packages/rs/deps/g3rs-deps-hook-contract`

Example TypeScript folders:

- `packages/ts/astro/setup/g3ts-astro-setup-hook-contract`
- `packages/ts/astro/content/g3ts-astro-content-hook-contract`
- `packages/ts/astro/mdx/g3ts-astro-mdx-hook-contract`
- `packages/ts/astro/seo/g3ts-astro-seo-hook-contract`
- future `packages/ts/style/g3ts-style-hook-contract`

Allowed ownership:

- `types` can define the hook requirement structs and static contract types.
- `hook-contract` owns the requirement-producing policy function for that family.
- ingestion can read external project facts for normal validation lanes, but it does not feed hook contracts until a concrete requirement proves that is unavoidable.

Forbidden ownership:

- ingestion must not publish policy.
- ingestion must not decide which validators are required.
- ingestion must not become the hook-requirement source of truth.
- config-checks must not define hook requirements.
- source-checks must not define hook requirements.
- file-tree-checks must not define hook requirements.
- types must not contain hook requirement logic. Types only define data structures.

Reason:

- ingestion consumes outside-world information.
- policy is authored by the family.
- checks verify already-ingested facts for their own lane.
- hook requirements are cross-lane family policy, so they need a separate family package.
- hooks consume policy plus parsed hook facts.

Contract shape:

```rust
pub fn hook_contract() -> Vec<G3HookRequirement>;

pub struct G3HookRequirement {
    pub id: String,
    pub owner_family: String,
    pub trigger_files: Vec<G3HookTriggerPattern>,
    pub required_commands: Vec<G3HookCommandRequirement>,
    pub fail_open_protected_commands: Vec<G3HookCriticalCommand>,
}
```

The exact type names can change, but the ownership must not:

- family owns what files/config changes require its validation
- family owns which validator command proves its policy
- hooks own whether the effective hook actually runs those commands
- hooks own shell safety and fail-open detection
- `hook_contract()` takes no parameters until a concrete requirement proves parameterization is unavoidable

Do not create two APIs for the same contract.

Forbidden:

```rust
pub fn static_hook_contract() -> Vec<G3HookRequirement>;
pub fn parameterized_hook_contract(input: G3HookContractInput) -> Vec<G3HookRequirement>;
```

If parameterization becomes unavoidable, change the single function intentionally:

```rust
pub fn hook_contract(input: G3HookContractInput) -> Vec<G3HookRequirement>;
```

That change must be justified by the first concrete rule that needs it.

Examples:

- `g3rs-clippy-hook-contract` declares: changes to `clippy.toml`, `Cargo.toml`, Rust source, and `rust-toolchain.toml` require a `cargo clippy ... -D warnings` command.
- `g3rs-deny-hook-contract` declares: changes to `deny.toml`, `Cargo.toml`, and `Cargo.lock` require `cargo deny check`.
- `g3rs-fmt-hook-contract` declares: changes to Rust source and `rustfmt.toml` require `cargo fmt --check`.
- `g3ts-astro-setup-hook-contract` declares: changes to Astro app source/config/content policy require app `validate`.
- `g3ts-astro-seo-hook-contract` declares: changes to Astro config, SEO helpers, crawler config, or artifact checker package config require app `validate`.
- future `g3ts-style-hook-contract` declares: changes to CSS/Tailwind/style config require style validation.

## Shared Hook Packages

Long-term shared packages should exist outside `packages/rs` and `packages/ts`:

- `packages/shared/hooks/g3-hooks-types`
- `packages/shared/hooks/g3-hooks-ingestion`
- `packages/shared/hooks/g3-hooks-file-tree-checks`
- `packages/shared/hooks/g3-hooks-source-checks`

These own:

- effective pre-commit selection
- `core.hooksPath`
- executable bit checks
- modular script inventory
- hook trust/shadowing checks
- shell safety checks
- `hook-shell-parser` usage

Language hook packages then own only aggregation and language-specific command equivalence:

- `packages/rs/hooks/g3rs-hooks-*`
- `packages/ts/hooks/g3ts-hooks-*`

Do not start by moving everything if that makes the first implementation too large. But do not duplicate shared logic blindly. If implementing G3TS hooks first, copy the G3RS behavior only as a stepping stone and mark the shared extraction as part of the same migration plan.

## Requirement Aggregation

Hook ingestion should produce hook facts:

- `G3EffectiveHookFacts`: selected hook, parsed scripts, executable metadata, trust risks.

The language hook orchestrator should collect requirements by calling family hook-contract packages:

```rust
let requirements = [
    g3rs_fmt_hook_contract::hook_contract(),
    g3rs_clippy_hook_contract::hook_contract(),
    g3rs_deny_hook_contract::hook_contract(),
]
.into_iter()
.flatten()
.collect::<Vec<_>>();
```

Hook checks should compare these.

This avoids hardcoding all command policy inside hooks.

Bad:

- `g3rs-hooks` has a hand-authored list of Rust config files.
- `g3ts-hooks` has a hand-authored list of Astro config files.

Good:

- `g3rs-clippy-hook-contract` declares Clippy hook requirements.
- `g3ts-astro-seo-hook-contract` declares Astro SEO hook requirements.
- hooks ingestion gathers effective hook facts.
- hooks orchestration gathers family hook contracts.
- hooks checks validate hook facts against hook contracts.

## Trigger Logic Must Improve Over Current G3RS

Current G3RS `config-changes-trigger-validation` is too weak because it checks whether hook logic mentions expected config paths.

The improved model:

- parse staged-file trigger branches into typed trigger facts when possible
- verify every declared trigger pattern reaches a required command
- fail closed when trigger logic cannot be parsed enough to prove coverage

Do not only check that strings like `clippy.toml` or `guardrail3-ts.toml` appear somewhere.

Acceptable first slice:

- keep current G3RS behavior unchanged
- add the family hook contract types and tests
- migrate one Rust family, for example `g3rs-fmt`, to declare requirements in `g3rs-fmt-hook-contract`
- make hooks consume that one declared requirement

Then migrate the rest family by family.

## Fail-Open Protection Must Use Declared Critical Commands

Current G3RS hardcodes critical commands in `no-fail-open-wrappers`.

Improved model:

- families declare critical commands they require
- hooks add universal critical commands such as `g3rs`, `g3ts`, `gitleaks`
- `no-fail-open-wrappers` checks the aggregate set

This matters for TS because Astro artifact checkers and future style validators are not globally known to hooks until the owning family opts into them.

## Package-Local Tool Resolution

G3RS can mostly check global tools because Rust cargo tools are normally binaries on `PATH`.

G3TS must not copy that assumption.

For TypeScript, hooks should understand:

- `pnpm exec <tool>`
- `pnpm run <script>`
- `pnpm --filter <package> run <script>`
- `npm run <script>`
- `yarn <script>`
- `bun run <script>`
- `node_modules/.bin/<tool>`

Tool availability checks should use package facts:

- if the hook uses `pnpm --filter landing run validate`, hooks check that the package exists and has a safe `validate` script
- the Astro family checks what `validate` contains
- hooks do not require global `astro`, `eslint`, or `syncpack`

## G3RS Migration Plan

G3RS should be improved before or alongside G3TS so Rust does not remain the worse legacy design.

Steps:

1. Add shared hook requirement types.
2. Add hook requirement aggregation to G3RS hooks orchestration from family-owned parameterless hook contracts.
3. Add one Rust family hook-contract package and teach it to declare requirements.
4. Replace the matching hardcoded G3RS hook rule with a requirement-driven rule.
5. Migrate remaining Rust families:
   - `fmt`
   - `clippy`
   - `deny`
   - `cargo`
   - `test`
   - `deps`
   - `code`
   - `garde`
   - `apparch`
   - `release`
6. Move shared hook structure/shell rules into shared hook packages.
7. Keep RS-specific command equivalence in G3RS hooks.

Do not remove old hardcoded G3RS hook rules until the matching family-declared requirement is enforced and tested.

# Shared Parser

Reuse:

- `packages/parsers/hook-shell-parser`

Do not implement a second TS hook parser.

- checking comments or `echo` text as proof

- `G3TsHooksSelectedHookConfigFact`
- `G3TsHookScriptKind`

- `has_tsconfig`

# Ingestion

`g3ts-hooks-ingestion` owns:

- deciding whether hooks scope is active
- reading `git config core.hooksPath`
- selecting the effective pre-commit surface
- reading `.githooks/pre-commit`
- reading `hooks/pre-commit` only as compatibility fallback
- reading direct `.githooks/pre-commit.d/*`
- collecting `.guardrail3/overrides/pre-commit.d/*`
- collecting hook trust risks such as `.husky/pre-commit`, Lefthook files, and shadowing `.git/hooks/pre-commit`
- parsing hook scripts with `hook-shell-parser`
- discovering installed tools from `PATH`
- discovering TS workspace/app facts from existing package parsers, not by hand-parsing package JSON

The effective hook selection should match G3RS:

- if `core.hooksPath = .githooks`, select `.githooks/pre-commit`
- if `core.hooksPath = hooks`, select `hooks/pre-commit`
- if unset, prefer `.githooks/pre-commit`, then `hooks/pre-commit`
- any other `core.hooksPath` is out of contract and file-tree checks should report it

The ingestion package must not decide rule pass/fail.

# Source Checks - TypeScript-Specific Commands

Add TypeScript-specific hook rules:

- `g3ts-hooks/g3ts-validate-staged-present`
- `g3ts-hooks/package-lint-step-present`
- `g3ts-hooks/ts-typecheck-step-present`
- `g3ts-hooks/eslint-step-present`
- `g3ts-hooks/test-step-present`
- `g3ts-hooks/duplicate-scan-step-present`
- `g3ts-hooks/ts-app-validate-step-present`
- `g3ts-hooks/ts-guardrail-config-changes-trigger-validation`
- `g3ts-hooks/package-config-changes-trigger-validation`
- `g3ts-hooks/lockfile-changes-trigger-package-validation`

Rule meanings:

- `g3ts-validate-staged-present`: effective pre-commit hook runs `g3ts validate --staged` or equivalent parsed command shape.
- `package-lint-step-present`: hook runs `syncpack lint` directly or through a reachable package script.
- `ts-typecheck-step-present`: hook runs a typecheck command directly or through configured app scripts. Accepted direct commands: `tsc --noEmit`, `astro check`.
- `eslint-step-present`: hook runs ESLint directly or through configured app scripts.
- `test-step-present`: hook runs package test command where TS packages declare test scripts.
- `duplicate-scan-step-present`: hook runs configured duplicate checker command when a TS duplicate policy exists.
- `ts-app-validate-step-present`: hook invokes the standard `validate` script for each configured TS app package that G3TS validates. This is the key bridge to Astro: Astro setup owns what `validate` must contain; hooks own that the hook calls it.
- `ts-guardrail-config-changes-trigger-validation`: changes to `guardrail3-ts.toml`, `eslint.config.*`, `.syncpackrc`, `tsconfig*.json`, Astro config, MDX/content config, and style config files must trigger TS validation.
- `package-config-changes-trigger-validation`: changes to `package.json`, workspace config, and package manager config must trigger package validation.
- `lockfile-changes-trigger-package-validation`: changes to `pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`, or `bun.lockb` must trigger package validation.

Do not enforce exact command spelling when the parser can prove an equivalent executable command.

Do not count comments, `echo`, or docs.

# Config Checks

G3TS hooks config checks should own installed-tool availability only when the hook actually requires the tool.

Rules:

- `g3ts-hooks/g3ts-binary-available`
- `g3ts-hooks/syncpack-binary-available`
- `g3ts-hooks/eslint-binary-available`
- `g3ts-hooks/typescript-binary-available`
- `g3ts-hooks/stylelint-binary-available`
- `g3ts-hooks/jscpd-binary-available`

Prefer local package binaries over global binaries when checking TS tools:

- `node_modules/.bin/<tool>`
- package manager execution through `pnpm exec <tool>`

Do not require globally installed `eslint`, `typescript`, or `syncpack` if the hook invokes them through package manager execution and the app/package dependency contract installs them.

`g3ts` itself is different:

- if the hook invokes `g3ts`, the binary must be available on `PATH` or invoked through a committed local package script that resolves it.

# Boundaries With Existing Families

Astro setup owns:

- `package.json` has `validate`
- `validate` runs ESLint, Syncpack, `astro check`, `astro build`, and artifact checkers
- Astro package pins and Astro forbidden deps

TS hooks owns:

- pre-commit hook invokes that app `validate` script
- hook does not hide failures
- hook triggers validation for config-only changes

Style family will own:

- what `lint:styles` means
- Stylelint/Tailwind package/config/script contract

TS hooks will own later:

- whether hooks invoke the style validation command once that family exists

Package/syncpack family owns:

- package version policy
- banned deps
- package manager constraints

TS hooks owns:

- whether package policy validation runs from the hook

# Rejected Designs

Rejected: put hook checks into Astro.

Reason:

- hook structure applies to all TS apps, not only Astro
- Astro already exposes `validate`; hook checks only prove the hook invokes it

Rejected: duplicate G3RS shared hook rules by copy-pasting raw string checks.

Reason:

- G3RS correctness comes from parser-backed executable command facts
- raw string checks recreate known false positives

Rejected: require one exact hook script layout beyond `.githooks/pre-commit` and optional `.githooks/pre-commit.d/*`.

Reason:

- G3RS already has a workable preferred-plus-compatibility contract
- forcing exact modular file names adds taxonomy without enforcement value
