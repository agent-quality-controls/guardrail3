## Goal

Build the first `ts/package` family wave for `g3ts`.

End state:

- a dedicated `package-json-parser` package
- `g3ts-package-types`
- `g3ts-package-ingestion`
- `g3ts-package-config-checks`
- `g3ts` wiring for `--family package`
- useful validation on real target roots without leaking package/tool ownership from sibling families

## Scope

Wave 1 stays narrow.

Root-only policy:

- root `package.json` exists
- root `package.json` parses
- root `private: true`
- root `packageManager` exists and is pinned to pnpm
- root `engines.node` exists
- root `engines.pnpm` exists
- root `scripts.preinstall` contains `only-allow pnpm`
- root `scripts.prepare` exists
- root `scripts.lint` exists
- root `scripts.typecheck` exists
- root `pnpm.overrides` exists
- root `pnpm.onlyBuiltDependencies` exists

Local-only policy:

- banned dependency declarations in `dependencies` and `devDependencies`

## Approach

1. Add a real `package-json-parser` package.
   - Use JSON parsing, not string scans.
   - Mirror the existing parser pattern: `Document { raw, typed }`.
2. Keep parser ownership narrow.
   - Parse manifest structure and selected fields.
   - Do not put policy inside the parser.
3. Keep family types narrow.
   - Convert parser output into a family-owned snapshot.
   - Do not leak parser documents through the family API.
4. Keep ingestion fail-closed.
   - Missing, unreadable, and parse-error states must be explicit.
   - Root and local applicability must be explicit.
5. Keep checks pure.
   - root policy rules operate only on normalized root facts
   - local banned-dependency rule operates only on one local manifest snapshot at a time

## Key Decisions

- Add a dedicated parser package instead of parsing directly in ingestion.
  - Why: `package.json` is a real file-format boundary and should match the parser seam already used for `eslint` and `tsconfig`.
- Keep tool-package presence out of `ts/package`.
  - Why: whether ESLint plugins are installed is not generic manifest policy. `ts/package` should not swallow sibling family ownership.
- Keep local manifest checks narrow in wave 1.
  - Why: the real immediate value on app/package roots is banned dependency declarations. The broader root script and engine policy is root-only.
- Require `engines.pnpm` in addition to `packageManager`.
  - Why: this is already part of the current local family contract, even though npm treats `engines` as advisory by default.

## Alternatives Considered

- Parse JSON directly inside `g3ts-package-ingestion`.
  - Rejected: same boundary mistake we already avoided in `tsconfig`.
- Check plugin/tool presence in `package.json`.
  - Rejected: effective ESLint evaluation is the stronger proof for ESLint. This family should stay on manifest policy.
- Apply root script rules to every local app/package manifest.
  - Rejected: the current family contract and target repo shape do not support that.
