## Goal

Create `packages/ts/eslint` as the first TS local-config family for ESLint validation.

End state for wave 1:

- discovers `eslint.config.*` surfaces using the TS root model
- validates existence and parseability
- validates the core typed-lint baseline
- validates the core plugin stack needed for that baseline
- stays out of `ts/hexarch`, `ts/css`, `ts/package`, and `ts/npmrc`

## Local Intent

From the current repo plans, `ts/eslint` is supposed to own:

- ESLint config existence
- ESLint config parseability
- core plugin presence
- plugin wiring
- baseline rule/severity checks

and explicitly not own:

- `ts/hexarch`
- `ts/css`
- `ts/tsconfig`

The current legacy code is boundary-mixed across:

- `eslint_check.rs`
- `eslint_plugin_checks.rs`
- `eslint_audit.rs`
- `eslint_parser.rs`
- `eslint_rule_infra.rs`
- plus package/config crossovers in `packages/config_files.rs` and `packages/package_deps.rs`

## External 2026 Delta

The old assumptions are stale in several important places:

- ESLint v10 now makes file-based config lookup the default and removes eslintrc.
- Flat config supports `basePath`, and `files` / `ignores` are relative to the config file or that base path.
- `eslint.config.ts/.mts/.cts` is now a real supported surface.
- `typescript-eslint` now recommends:
  - `projectService: true`
  - `recommendedTypeChecked`
  - `strictTypeChecked`
  - `stylisticTypeChecked`
  - `disableTypeChecked` for JS carve-outs
  - `defineConfig()` over the older helper shape

These are strong reasons not to clone the legacy checker behavior blindly.

## Approach

1. Treat ESLint configs as typed local config files, not grep targets.
2. Give the family a root-aware orchestrator that discovers relevant `eslint.config.*` files.
3. Parse once into a typed ESLint config model that is good enough for rule presence, plugin wiring, and top-level composition checks.
4. Fan out into small pure rules.
5. Read `package.json` as a typed cross-family input only where package-presence checks are truly part of `ts/eslint`.

## Wave 1 Rule Cut

Wave 1 should include only the durable baseline:

- config exists
- config parses
- config uses flat config shape
- config uses modern `typescript-eslint` typed linting setup
- config enables `projectService: true`
- config includes the chosen typed presets
- config has explicit JS/test carve-outs where typed linting is disabled
- config includes the core plugin stack needed by the lint baseline

Core package presence in wave 1:

- `@eslint/js`
- `typescript-eslint`
- `eslint-plugin-unicorn`
- `eslint-plugin-regexp`
- `eslint-plugin-sonarjs`
- `eslint-plugin-react`
- `eslint-plugin-react-hooks`

Wave 1 should not include:

- CSS-specific policy
- architecture-specific boundary policy
- package-manager policy
- root package manifest policy
- legacy substring-matching checks

## Key Decisions

- Do not inherit the legacy string-match checker architecture.
  - Reason: the historical audits already proved it is easy to bypass and easy to mis-read.

- Treat root-vs-local config ownership as a first-class decision.
  - Reason: ESLint v10 lookup behavior changes the shape of monorepo enforcement.

- Keep package-presence checks minimal.
  - Reason: `ts/eslint` should validate the lint toolchain it directly needs, not every TS dev tool.

## Latest Ideas, Not Final Decisions

- The canonical ESLint config may move to `eslint.config.ts`, not `eslint.config.mjs`.
- `projectService: true` should likely be the default typed-lint posture from wave 1.
- The right first boundary may be one root config plus optional local configs, not per-app independent configs everywhere.
- `@next/eslint-plugin-next` probably belongs in a later content/app-specific layer, not the first generic `ts/eslint` wave.

## Risks / Open Questions

- `strictTypeChecked` and `stylisticTypeChecked` are not semver-stable preset names, so version churn handling must be explicit.
- `projectService` makes tsconfig coverage stricter, so JS/scripts/test carve-outs need to be validated intentionally.
- We still need to choose the exact typed parser strategy for ESLint flat config in Rust.
