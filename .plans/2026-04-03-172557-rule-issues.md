# Rule Issues

Open issues found during the rule message audit. Each needs design work or implementation.

---

## Redesign: CLIPPY-14 global state bans

**Problem:** CLIPPY-14 bans `LazyLock`, `OnceLock`, `once_cell::Lazy`, `once_cell::OnceCell` in `disallowed-types`. Currently gated on `profile_name == "library"` which no longer exists. The rule checks at the clippy.toml level (config), not at the code level.

**Why it matters:** Global state in libraries forces state on consumers. In apps, global state (config singletons, static regex, connection pools) is legitimate.

**What needs to happen:** The rule needs to determine whether a workspace is a library or an app. A workspace with `main.rs` is an app; without `main.rs` it's a library. Libraries should warn about global state types; apps should skip the check.

**Current blockers:**
- No existing mechanism in the legality/placement layer to classify a workspace as app vs library based on `main.rs` presence
- The rule operates at clippy.toml config level, not per-crate — it can't easily distinguish per-crate
- No escape-hatch support — libraries that legitimately need `OnceLock` have no opt-out
- Should be Warn, not Error, since there are legitimate exceptions

**Recommendation:** Redesign completely. Add workspace-level app/library detection based on `main.rs` presence. Make it Warn. Add escape-hatch support.

## Move: CARGO-15 rust-version → Release family

**Problem:** CARGO-15 requires `rust-version` (MSRV) but is gated on library profile. MSRV is a publishing concern — only matters for crates consumers depend on.

**What needs to happen:** Move this rule to the release family alongside PUB-01 through PUB-07. Gate on `publishable` instead of `profile_name == "library"`. The release family already has per-publishable-crate inputs (`PublishableCrateReleaseInput`).

## Remove: TOPOLOGY-06 Owner Family Enablement Coherence

**Problem:** References the retired package-only family. The logic is wrong: having a crate under `apps/` should not force enabling hexarch checks. The rule fires when a crate is in a topology zone but the zone's enforcement family is disabled - but that's a valid configuration (you can have apps without hexarch).

**Action:** Remove the rule entirely. The check function, inputs, facts, and test files all need cleanup.

## Split: Rules that check multiple things

These rules should each check one thing. Splitting reduces cascade noise and makes filtering/suppression granular.

### ARCH-02: lib.rs facade-only
1. Implementation logic in lib.rs (body_items)
2. Broad re-exports in lib.rs (glob `pub use`)

### ARCH-04: mod.rs facade-only
1. Implementation logic in mod.rs (body_items)
2. Broad re-exports in mod.rs (glob `pub use`)

### ARCH-08: feature-gated exports
6 checks in one rule:
1. Ungated pub exports in lib.rs
2. Exports gated directly on `all` instead of named features
3. Missing `all` feature in Cargo.toml
4. Empty `all` feature
5. Missing `default` feature
6. `default` doesn't include `all`

A crate with no feature gates triggers 1, 3, and 5 simultaneously — cascade noise.

### TOOLCHAIN-02: Channel and Components
1. TOML parse validity
2. Channel policy (stable/pinned required)
3. Components policy (clippy, rustfmt required)

### TOPOLOGY-12: Declared Workspace Members Only
1. Undeclared workspace member (crate exists but not in members list)
2. Extra workspace member (in members list but no matching crate)

These are opposite violations.

## Consolidate: Redundant/overlapping rules

### TOPOLOGY-01, 03, 04: Zone overlap
All three check the same underlying problem from different angles:
- 01: per-crate symptom (ambiguous classification)
- 03: per-crate symptom (dual ownership)
- 04: zone-level cause (zones physically nest)

The fix is always "restructure so zones don't overlap." Consider merging into one rule.

### TOPOLOGY-09, 10: Top-level workspace
- 09: top-level root must declare `[workspace]`
- 10: top-level root has `[package]` but no `[workspace]`

10 is a subset of 09. Both fire on the same underlying problem.

## Rename: Release family ID prefix unification

The release family uses three different ID prefixes:
- `RS-RELEASE-*` (repo policy + inventory)
- `RS-PUB-*` (publish metadata + integrity)
- `RS-BIN-*` (binary release)

All should be `RS-RELEASE-*` for consistency. ~39 files affected (rules, assertions, tests).
