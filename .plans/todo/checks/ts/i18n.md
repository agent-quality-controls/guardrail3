# TS-I18N — Locale routing and message completeness checker

**Input:** discovered TS apps, locale/message files, routing config, i18n-related source/config files
**Parser:** directory discovery + JSON + targeted TS/TSX/config inspection
**Current code:** `app/ts/validate/i18n_check.rs`
**Owned root:** TS package/app roots with i18n enabled by config or auto-detection

## Owns

- i18n library/config detection
- locale routing configuration
- locale/message file presence
- locale/message completeness and parity
- message-key consistency within the supported contract
- translation-usage correctness in i18n-enabled apps
  - no hardcoded locale-path literals when locale helpers are required
  - no user-visible string bypass of the translation surface where the app contract requires translations

## Does not own

- content pipeline/model structure
  - that belongs to `ts/content`
- sitemap, robots, metadata, structured data
  - that belongs to `ts/seo`
- generic source-scan hardcoded-string policing
  - that belongs to `ts/code`

## Contract direction

This family should be independent of content-site modeling.

Content sites often need it, but i18n is not intrinsically a content-only concern.

It should own:
- locale routing
- locale/message artifacts
- message completeness
- translation-surface correctness for i18n-enabled apps

and nothing broader.
