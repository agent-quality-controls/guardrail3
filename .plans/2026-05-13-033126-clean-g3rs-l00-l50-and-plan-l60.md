# Clean G3RS L00-L50 Fixtures And Plan L60

## Goal

Clean G3RS behavior replay fixtures through L50 so every visible low-layer behavior is represented without hidden Error/Warn rows.

Add a concrete L60 plan for delegated-policy-invalid behavior after delegated tools are present.

## Approach

- Update the existing behavior manifest `.plans/2026-05-12-183156-guardrail3-behavior-replay-fixture-migration.md.manifest.toml`.
- Update fixture files under `behavior/fixtures/g3rs`.
- Regenerate generated baselines under `behavior/baselines/g3rs`.
- Keep generated JSON baselines generated by `scripts/behavior/generate-baselines.py`; do not hand-edit baseline JSON.
- Verify with `scripts/behavior/verify-all.sh`.

## L00 Cleanup

- Extend `L00-workspace-root-not-found` with a nested Cargo workspace under `repo/apps/service`.
- Keep the command pointed at `repo`.
- Expected behavior: `g3rs validate --path .` still reports missing root Cargo.toml for the selected root and does not auto-select `repo/apps/service`.
- This does not hide an existing L00 behavior because L00 has no Error/Warn rows and the failure is stderr/root-selection behavior.

## L40 Cleanup

- Add malformed `.cargo/config.toml` to `L40-required-inputs-present-invalid`.
- Merge `L42-member-cargo-toml-invalid` into `L40-required-inputs-present-invalid`.
- Delete the separate `L42-member-cargo-toml-invalid` fixture and baseline.
- Reason: adversarial merged run showed all existing L40 and L42 Error/Warn rows remain visible together.
- Remove no-effect fixture files from `L41-root-cargo-toml-invalid`: `repo/Cargo.lock` and `repo/src/lib.rs`.

## New L40 Split Fixtures

- Add `L43-fmt-nightly-key-toolchain-invalid`.
- It uses a parseable rustfmt config with a nightly-only key and a malformed `rust-toolchain.toml`.
- Expected behavior: `g3rs-fmt/nightly-keys-on-stable` reports the toolchain parse failure.
- Reason for split: the base L40 fixture has malformed `rustfmt.toml`, which hides this rule branch.

- Add `L44-clippy-typed-config-invalid`.
- It uses a TOML-parseable but typed-invalid `clippy.toml`.
- Expected behavior: `g3rs-clippy/config-parseable` reports a clippy config parse error.
- Reason for split: one `clippy.toml` cannot be both raw TOML-invalid and typed-invalid.

- Do not add a release README source-input fixture at this layer.
- Reason: the public `g3rs validate --family release` boundary cannot isolate README source-input failure without also activating `g3rs-release/publish-dry-run`.
- Keeping that as L40 would mix invalid-input coverage with delegated publish behavior.

## L50 Cleanup

- Add `L50-test-cargo-mutants-missing`.
- Activate mutation policy with valid `.cargo/mutants.toml`, `profile.mutants`, and mutation hook wiring.
- Hide `cargo-mutants` from the runner PATH the same way L50 hides other delegated tools.
- Expected behavior: `g3rs-test/cargo-mutants-installed` reports `cargo-mutants missing`.
- Reason for split: the existing L50 fixture covers deps-family delegated tools; this one covers the test-family delegated tool under a different activation state.

## L60 Plan

L60 means required inputs are valid and delegated tools are present, but delegated policy wiring is weakened or wrong.

Build L60 as separate fixtures only when one policy failure hides another policy failure. Do not split by package or by old unit test.

Candidate L60 checks:

- Existing `L60-delegated-tools-present-policy-invalid` keeps the clippy policy wiring failure:
  - `g3rs-clippy/too-many-lines-threshold`
  - input: valid `clippy.toml` with wrong threshold
  - reason: delegated Clippy is present but policy is weakened

- Add L60 cargo-deny policy fixture if current tests show a visible rule after tools are present:
  - candidate rule area: deny config semantic policy, not missing `deny.toml`
  - expected shape: valid TOML with missing required ban baseline, wrong bans, wrong license/source policy, or shadowed allow policy
  - reject if the behavior is already L40 malformed input or L50 missing `cargo-deny`

- Add L60 cargo-machete policy fixture only if G3RS has a delegated-policy check beyond tool presence:
  - expected shape: tool present, command output indicates policy weakening
  - reject if current implementation only checks install presence

- Add L60 cargo-dupes policy fixture only if G3RS has a delegated-policy check beyond tool presence:
  - expected shape: tool present, duplicate policy output violates configured threshold
  - reject if current implementation only checks install presence

- Add L60 gitleaks policy fixture only if G3RS has a delegated-policy check beyond tool presence:
  - expected shape: tool present, gitleaks config or invocation is weakened
  - reject if current implementation only checks install presence

- Add L60 mutation policy fixture if current tests show policy wiring failures after `cargo-mutants` is present:
  - candidate rule area: `.cargo/mutants.toml` sane configuration or hook command weakened
  - reject if the behavior is L30 missing input, L40 malformed input, or L50 missing `cargo-mutants`
