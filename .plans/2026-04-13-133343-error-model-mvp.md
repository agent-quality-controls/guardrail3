# Error Model - MVP

## Goal

- Define one consistent error model across `web`, `backend`, and `crawler`
- Avoid ad hoc error shapes per route or service
- Keep expected product failures separate from unexpected system failures

## Decisions

### 1. HTTP service boundaries use Problem Details

- `backend` and `crawler` return HTTP errors as `application/problem+json`
- The response shape follows RFC 9457 Problem Details
- Websmasher adds a small set of app-specific extension fields

Minimum shape:

```json
{
  "type": "https://errors.websmasher.com/auth/no-org",
  "title": "No organization available",
  "status": 403,
  "detail": "The authenticated user does not have an active organization.",
  "code": "auth.no_org",
  "retryable": false
}
```

### 2. Internal code uses typed error categories

- Services should classify errors into a small set of categories:
  - `validation`
  - `authentication`
  - `authorization`
  - `not_found`
  - `conflict`
  - `rate_limited`
  - `upstream`
  - `timeout`
  - `internal`

- These are design categories, not necessarily public wire values

### 3. Expected errors are returned explicitly

- Expected errors are normal product outcomes
- They must be converted into structured responses
- They must not be treated as crashes

Examples:
- invalid input
- authenticated user has no org
- multi-org unsupported in MVP
- crawl job not found
- report not ready yet
- conflict with current resource state

### 4. Unexpected errors are generic at the boundary

- Unexpected errors are bugs, broken invariants, uncaught dependency failures, or unknown failure modes
- They should be logged with full diagnostics
- They should return a generic internal error response at the service boundary
- Clients must not receive stack traces or internal implementation details

### 5. `web` handles expected errors as values

- In normal UI flows, `web` should return structured expected failures to the UI
- `web` should not rely on thrown exceptions for ordinary product failures
- Throwing is reserved for unexpected failures that should hit framework error boundaries

### 6. Every boundary error has a stable `code`

- Errors exposed across service boundaries must have a stable low-cardinality code
- Codes are the main machine-readable identifier for product and integration logic

Examples:
- `auth.no_org`
- `auth.multi_org_unsupported`
- `auth.provisioning_failed`
- `validation.invalid_input`
- `crawl.job_not_found`
- `crawl.job_already_running`
- `report.not_ready`
- `report.not_found`
- `internal.unexpected`

### 7. Retryability is explicit

- Retryability is not inferred from status code alone
- If an error is safe or reasonable to retry, the error payload should say so explicitly
- Validation, auth, and authorization failures are normally not retryable
- Timeouts and some upstream failures may be retryable

## Problem Details extensions

Websmasher may extend Problem Details with:
- `code`
- `retryable`
- `field_errors`

Example validation error:

```json
{
  "type": "https://errors.websmasher.com/validation/invalid-input",
  "title": "Invalid input",
  "status": 400,
  "detail": "One or more fields are invalid.",
  "code": "validation.invalid_input",
  "retryable": false,
  "field_errors": {
    "url": ["Must be a valid URL"]
  }
}
```

## Logging rules

- Unexpected failures are logged as errors with full diagnostics
- Expected product failures are logged at lower severity when logging is useful
- Do not log expected validation failures as if they were production incidents
- Do not send stack traces or secret-bearing diagnostics to clients

## Observability rules

- Error `code` should be the main low-cardinality identifier used across:
  - logs
  - traces
  - metrics
- Trace and request correlation metadata may travel separately from the error payload
- Observability metadata is not part of authorization or product semantics

## Status code guidance

- `400` - invalid request input
- `401` - unauthenticated
- `403` - authenticated but not allowed, or missing required org context
- `404` - resource not found
- `409` - state conflict
- `429` - rate limited
- `502` - upstream dependency returned a bad response
- `504` - upstream timeout
- `500` - unexpected internal failure

## What this does not require

- One giant shared implementation library across all services
- Rich error inheritance trees everywhere
- A complete final list of all error codes today

## What this does require

- One canonical HTTP error format
- One shared naming style for codes
- One clear split between expected and unexpected failures
- Safe client-facing responses

## Summary

- HTTP APIs return Problem Details
- Websmasher adds `code`, `retryable`, and `field_errors`
- Expected failures are explicit structured outcomes
- Unexpected failures are generic to clients and detailed in logs
- Stable error codes become the shared cross-service contract
