## Summary

Built `ts/apparch` wave 1 and wired it into `g3ts` as a structure family for one explicit Next-style app root. The family mirrors the live Rust `apparch` seam with `types`, `ingestion`, `config-checks`, and `source-checks`, but adapts the dependency graph from Rust crate edges to app-internal TS import edges under `src/types`, `src/logic`, `src/io/inbound`, `src/io/outbound`, and `src/app`.

## Decisions made

- Replaced the earlier package-graph `apparch` idea with app-internal layer roots.
  - Why: the user clarified that `apparch` is for app structure, not workspace packages.
  - Rejected: package-root dependency analysis, because that belongs to a different family.

- Kept wave 1 to `config` and `source` lanes only.
  - Why: this matches the live Rust `apparch` seam more closely than inventing a fake file-tree lane.
  - Rejected: copying `ts/arch`'s file-tree lane into `apparch`.

- Built config facts from source-file imports, not `package.json`.
  - Why: a single Next app's real dependency graph is its internal import graph.
  - Rejected: manifest dependency edges, because they do not represent app-internal architecture.

- Fixed the parser boundary bug in ingestion instead of weakening the checks.
  - Added `DynamicImport` to `G3TsApparchImportKind`.
  - Added recursive dynamic-import extraction.
  - Added exported function-valued binding extraction from `export const ... = () => ...`.
  - Why: both were fact-generation bugs that would silently miss real apparch violations.

- Kept `src/app` as a thin framework-owned shell by rule direction, not by moving files out of `app`.
  - `app -> io/outbound` is forbidden.
  - Why: Next requires route and page entry files to stay in `app`.
