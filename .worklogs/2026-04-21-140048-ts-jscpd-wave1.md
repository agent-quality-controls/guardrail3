## Summary

Built the `ts/jscpd` wave-1 family and wired it into `g3ts`. The new family now parses root `.jscpd.json`, converts it into narrowed family snapshots, enforces the root baseline, and validates cleanly against the live `websmasher` repo root.

## Decisions made

- Kept `ts/jscpd` root-only for the explicit target root.
  - Why: wave 1 is about the root duplication-policy surface, not nested content policy.
  - Rejected: copying legacy content checks into this family, because those belong to a future `ts/content` family.

- Kept the parser JSON-only and parser-owned.
  - Why: `.jscpd.json` is data, not executable config. A dedicated JSON parser package is the correct boundary.
  - Rejected: folding parse logic into ingestion, because that would leak parser semantics through the family boundary.

- Removed stale copied `npmrc` debris instead of carrying it under waivers.
  - Why: the copied extra rule files and stale metadata were not architecture, just scaffold debt.

- Treated the long `g3ts validate --family jscpd` repo-root run as a real verification concern.
  - Why: the new family had to prove itself on the live `websmasher` root, not only in unit fixtures.
  - Result: the run completed cleanly; the cost was in the shared workspace crawl over the large target repo, not a `jscpd` logic loop.
