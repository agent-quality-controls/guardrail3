## Summary
Removed direct rule-result semantic assertions from the remaining targeted sidecars by replacing direct `errors_by_id`/field checks with owned assertion helper calls, plus helper-based ownership split checks for rules 09/10 interactions.

## Context & Problem
RS-TEST-16 was still firing in the hexarch `rules 05-10` sidecars. The target was to keep sidecars as fixture setup plus helper-call-only semantics and avoid direct assertions against `CheckResult` fields/rule matching logic.

## Decisions Made

### Scope-limited remediation for rules 05-10
- **Chose:** Refactor only files under rules `05-10` runtime sidecars (and only touched assertions modules already in scope when needed).
- **Why:** User explicitly requested this range and forbidden editing outside those rules and `assertions_common`.
- **Alternatives considered:**
  - Rewriting `assert_expected_rule_results` to a backward-compatible signature for all rules (rejected: broader blast radius and out-of-scope).
  - Disabling RS-TEST-16 warnings globally in validation (rejected: does not address actual test quality regression).

### Preserve file-level API constraints
- **Chose:** Use explicit helper calls (`assert_no_error`, `assert_error_count`, `assert_expected_rule_results`, `assert_any_result_contains_title`) with rule IDs passed into helper calls where needed.
- **Why:** Keeps semantic checks in one place while still allowing cross-rule ownership checks (e.g., “rule 09 owns X not rule 10”).
- **Alternatives considered:**
  - Keeping manual `errors_by_id` with `len`/`title.contains` checks (rejected by RS-TEST-16 design intent).
