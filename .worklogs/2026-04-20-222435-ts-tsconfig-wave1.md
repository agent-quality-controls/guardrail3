Summary

Built the first `ts/tsconfig` family wave plus a dedicated `tsconfig-json-parser` and wired the family into `g3ts validate`. The new slice validates clean under `g3rs`, and `g3ts` now validates the `web` and `landing` app roots with `--family tsconfig`.

Decisions made

- Added a dedicated `packages/parsers/tsconfig-json-parser` package instead of parsing inside ingestion.
  - Why: `tsconfig.json` is a real file format boundary and the family should consume a parsed document, not raw JSON or ad hoc maps.
  - Rejected: parsing JSON directly inside `g3ts-tsconfig-ingestion`, because that would blur parser and family responsibilities.
- Anchored ingestion on the explicit root `tsconfig.json` for the validated app/package root.
  - Why: `g3ts` is validating TS app roots directly, so repo-global `tsconfig.base.json` discovery is the wrong ownership boundary.
  - Rejected: repo-root-first discovery, because it would make app-local validation depend on unrelated parent crawl state.
- Resolved local `extends` chains in ingestion and computed effective compiler options before rules run.
  - Why: config checks should read normalized facts, not perform traversal or merge logic.
  - Rejected: letting each rule walk the `extends` chain independently.
- Allowed external package-style `extends` entries in wave 1 and deferred full external-proof enforcement.
  - Why: app roots commonly extend shared package configs outside the local crawl, and wave 1 needs to stay useful on real repos.
  - Rejected: failing all external extends immediately, because it would make the family unusable on normal monorepo setups.
- Enforced a strict 12-flag baseline in wave 1.
  - Why: this matches the current plan and gives `g3ts` immediate value on real TS app roots.
  - Rejected: a looser partial baseline, because it would under-spec the family.
