# Merge RS-CODE-28 Into RS-CODE-27

**Date:** 2026-03-27 18:38
**Scope:** `.plans/todo/checks/rs/code.md`, `apps/guardrail3/crates/app/rs/families/code/crates/runtime/src/lib.rs`, `apps/guardrail3/crates/app/rs/families/code/crates/runtime/src/parse.rs`, `apps/guardrail3/crates/app/rs/families/code/crates/runtime/src/parse/visitors.rs`, `apps/guardrail3/crates/app/rs/families/code/crates/runtime/src/rs_code_27_facade_only_lib_tests/*`, removed `rs_code_28_*`, `apps/guardrail3/crates/app/rs/families/code/crates/assertions/src/lib.rs`

## Summary
Collapsed the old `RS-CODE-28` warning into `RS-CODE-27` so `lib.rs` now has one strict facade rule: any inline module body in a library `lib.rs` is an error. Removed the duplicate rule implementation, migrated coverage into `RS-CODE-27`, and updated the rule inventory to make the stricter contract explicit.

## Context & Problem
`RS-CODE-27` and `RS-CODE-28` had drifted into a contradictory contract. The plan said `lib.rs` should stay facade-only, but the implementation still tolerated private inline modules and only warned on public inline modules under a second rule. That split was unnecessary and made the policy ambiguous. The decision from the review was to enforce the stricter architectural meaning: `lib.rs` should contain declarations only, not module bodies.

## Decisions Made

### Merge inline-module enforcement into RS-CODE-27
- **Chose:** Treat any inline module body in library `lib.rs` as an `RS-CODE-27` error.
- **Why:** This matches the “facade-only” contract directly and removes the policy split between “bodies are forbidden” and “some bodies are only warnings.”
- **Alternatives considered:**
  - Keep `RS-CODE-28` as a warning for public inline modules only — rejected because it preserved the ambiguity and still allowed private inline modules despite the facade contract.
  - Keep both rules but escalate `RS-CODE-28` to error — rejected because it still duplicated rule ownership for the same structural problem.

### Keep the rule structural, not context-sensitive
- **Chose:** Inline module bodies under `#[cfg(test)]` are also forbidden in library `lib.rs`.
- **Why:** Test-only inline modules are still bodies in `lib.rs`, and exempting them would recreate the same facade leak under a special case.
- **Alternatives considered:**
  - Exempt `#[cfg(test)] mod tests` — rejected because it weakens the facade rule and overlaps with `RS-TEST` style concerns instead of preserving a clear source-shape rule.

### Remove the dead rule surface completely
- **Chose:** Delete the old `RS-CODE-28` runtime rule, assertions module, and sidecar tests rather than leaving them retired-but-live.
- **Why:** Keeping dead rule files around would confuse future audits and can trip `RS-TEST` on stale sidecar directories.
- **Alternatives considered:**
  - Leave the files in place but stop wiring them — rejected because it leaves stale rule surfaces in the family and caused an immediate `RS-TEST-02` failure on the empty sidecar directory.

## Architectural Notes
This change makes `RS-CODE` more consistent with the stabilized family pattern: one structural concern, one rule owner. `RS-CODE-27` is now the sole owner of `lib.rs` facade enforcement. The parser remains purely structural: it identifies inline module bodies as one more forbidden item kind, alongside functions, impls, private `use`, macro items, and extern blocks.
