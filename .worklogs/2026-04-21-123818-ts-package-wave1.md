## Summary

Built `packages/ts/package` wave 1 and wired it into `apps/guardrail3-ts` as the new `package` family. The family now parses `package.json`, separates workspace-root policy from local manifest policy, and keeps the public family input narrowed to family-owned snapshots rather than parser documents.

## Decisions made

- Added a dedicated `package-json-parser` package instead of parsing inside family ingestion.
  - Why: matches the repo parser pattern and keeps JSON parsing separate from package-policy semantics.
  - Rejected: hand-rolled `serde_json` parsing inside `g3ts-package-ingestion`, because that would collapse parser and family boundaries immediately.

- Split `package` applicability into package-manager root policy and local manifest policy.
  - Why: root rules like `private`, `packageManager`, `engines`, scripts, and `pnpm.*` do not apply to ordinary app manifests.
  - Rejected: treating any explicit target with `package.json` as a workspace root, because that broke `apps/landing` and other leaf targets.

- Kept the family API narrow.
  - Why: `g3ts-package-types` now exposes family snapshots, not parser documents, which keeps `config-checks` from depending on parser internals.
  - Rejected: exposing `PackageJsonDocument` directly through family state, because that would repeat the earlier TS boundary mistakes.

- Tightened `TS-PACKAGE-CONFIG-08` to stay generic.
  - Why: banning `pg` and `postgres` in a reusable manifest family was too project-specific. The family should catch generic banned manifest dependencies, not hardcode one app's outbound adapter policy.
  - Rejected: keeping `pg` banned and forcing target repos to contort around a generic family that should not own database-driver architecture.
