## Testing Process (follow this exactly)

### For each rule:

1. **Write exhaustive tests** — send an agent with the check code + golden fixture + truth table. Tests must:
   - Break ALL locations where possible (not just one app)
   - Assert exact error count (`assert_eq!`, never `>=` or `!is_empty()`)
   - Assert title content (violation type + specific content)
   - Assert file field on `&r6` (rule-specific errors, NOT `&errors`)
   - Assert per-app attribution on `&r6`
   - Assert inner hex label in title
   - Assert `assert_no_ts_apps(&r6)` and `assert_no_packages(&r6)`
   - Assert message content
   - Also break same structure in TS apps, assert 0 RS errors

2. **Run /test-attack** — the skill orchestrates:
   - Phase 0: Reconstruct intent from plan + code + golden. Write truth table. Check for CHECK BUGs.
   - Launch 3 agents: completeness (per-test PASS/FAIL), missing scenarios + branch coverage, false positives
   - Collect unified report

3. **Fix all findings** — test assertion gaps AND check code bugs

4. **Run /test-attack again** — verify fixes, find new gaps from the fixes

5. **Repeat until converged** — agents find 0 issues

### Key assertion patterns

```rust
// Filter to rule-specific errors (not all arch errors)
let r6 = rule6_errors(&errors);

// All assertions on &r6, NOT &errors
assert_eq!(r6.len(), EXACT_COUNT, "...");
assert_per_app(&r6);        // each Rust app appears
assert_inner_hex(&r6);      // at least one from mcp/crates
assert_file_field(&r6);     // every error has file set
assert_no_ts_apps(&r6);     // no admin/landing
assert_no_packages(&r6);    // no shared-types/ui-kit

// Title + message content per error
for err in &r6 {
    assert!(err.title.contains("specific violation"), "...");
    assert!(err.message.contains("specific detail"), "...");
}
```

### Design auditor pattern

Include a "design auditor" agent that reads the PLAN (not just the code) and asks: "does the code implement the intended behavior?" This catches bugs that tests accidentally validate.

## Key Learnings

1. **Tests that pass are suspicious** — if all tests pass after writing, the tests may validate buggy behavior. Always include a design auditor that compares code to INTENT.
2. **Assertions on &r6, not &errors** — filtering to rule-specific errors before asserting prevents other rules from satisfying per-app/inner-hex/file-field checks vacuously.
3. **assert_eq! always, never >=** — even for negative tests, use `assert_eq!(len, 0)` not `assert!(is_empty())`.
4. **Every test breaks EVERYWHERE** — not one app, ALL apps + inner hex simultaneously.
5. **False positive checks in every test** — TS apps, packages, unbroken apps must all be verified NOT flagged.
6. **Message content matters** — the check produces specific messages; tests should verify them.
7. **CHECK BUGS take priority** — fix the code before touching tests. A test that validates buggy behavior is worse than no test.
