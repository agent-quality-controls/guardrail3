Summary

Built `ts/arch` wave 1 and wired it into `g3ts` as the first TS structure family. The new family mirrors the live Rust `arch` seam with `types`, `ingestion`, `config-checks`, `source-checks`, and `file-tree-checks`, and the app now routes `arch` through its own bounded structure runner group.

Decisions made

- Mirrored the live Rust split instead of stale TS plans.
  - `g3ts-arch-types` owns the family contract.
  - `g3ts-arch-ingestion` owns root manifest discovery, facade parsing, and source-tree metrics.
  - `g3ts-arch-config-checks`, `g3ts-arch-source-checks`, and `g3ts-arch-file-tree-checks` own pure rules.
- Used package-manifest-declared public entrypoints as the TS facade analogue for wave 1.
  - Current declared entrypoint sources:
    - `package.json#types`
    - `package.json#exports["."]`
- Kept the structure runner separate from the config runner in `g3ts`.
  - This mirrors `g3rs` runner grouping instead of stuffing `arch` into the config group.
- Fixed the first file-tree modeling bug at the family boundary.
  - File-tree input now carries exact `existing_entrypoints` rather than guessing from `source_tree.is_some()`.
- Removed copied dead `crates/types` roots from the check packages instead of leaving them as orphan workspace children.
- Reduced ingestion source parsing from one oversized string `match` to smaller typed helpers and moved direct file reading behind a dedicated `fs` module.
