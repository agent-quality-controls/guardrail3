## Summary

Built `packages/ts/npmrc` wave 1 and wired it into `g3ts` as the new `npmrc` family. The family now treats `.npmrc` as a root-only, line-oriented config surface with parser-owned duplicate-key and last-wins semantics, while keeping the public family input narrowed to one root snapshot.

## Decisions made

- Added a dedicated `npmrc-parser` package.
  - Why: `.npmrc` parsing concerns are real and parser-specific - BOM stripping, inline comments, quoted empty values, duplicate keys, and last-wins lookup all belong below family ingestion.
  - Rejected: parsing `.npmrc` ad hoc inside `g3ts-npmrc-ingestion`, because that would collapse parser and family boundaries.

- Kept the family root-only.
  - Why: local planning and current pnpm usage point to one package-manager-root `.npmrc`, not per-app enforcement.
  - Rejected: copied local-manifest handling from `ts/package`, because it would be the wrong ownership model for this file.

- Kept wave 1 narrow.
  - Why: the legacy 13-key baseline is too broad for a durable 2026 `.npmrc` family.
  - Chosen required baseline:
    - `strict-peer-dependencies=true`
    - `disallow-workspace-cycles=true`
    - `engine-strict=true`
    - `minimum-release-age=1440`
    - `block-exotic-subdeps=true`
    - `trust-policy=warn`
  - Everything else is inventory, not error, in wave 1.

- Preserved duplicate-key visibility and last-wins semantics together.
  - Why: duplicate keys are always suspicious, but the effective-value calculation still has to match pnpm behavior.
