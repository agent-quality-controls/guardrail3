# RS-TEST-16 cleanup in hexarch 11-25 and dependency_facts assertions migration

**Date:** 2026-03-26 20:15:40
**Scope:** `apps/guardrail3/crates/app/rs/families/hexarch/crates/runtime/src/*_tests`, `apps/guardrail3/crates/app/rs/families/hexarch/crates/assertions/src/*` for rules 11-25 plus dependency_facts

## Summary
Migrated remaining test semantic checks for hexarch rules 11–25 (and dependency_facts) from runtime sidecars into owned assertions modules, and left sidecars as setup + assertion-helper calls. Ran requested validation commands and confirmed RS-TEST-16 is no longer reported for this scope.

## Context & Problem
The runtime-side tests for rules 11–25 still contained direct semantic assertions (field/title/message/file/rule-id checks) and some assertion helpers in `dependency_facts_tests` were inline. The request was to move those checks into owned assertions modules and keep sidecars as setup-only wiring.

## Decisions Made

### Assertion migration for rule 11–25
- **Chose:** add rule-specific assertion helpers in each owned `rs_hexarch_XX_*` assertions module and update matching test files to call those helpers.
- **Why:** preserves per-rule ownership and satisfies RS-TEST-16 expectations that proof-site logic lives in owned assertion modules.
- **Alternatives considered:**
  - Keeping direct checks in sidecars and broadening RS-TEST suppression — rejected because it contradicts RS-TEST-16 intent and weakens proof-site guarantees.
  - Moving checks into shared/common helpers — rejected because rule modules already own the semantics and need clearer per-rule ownership boundaries.

### `dependency_facts` assertion helpers
- **Chose:** replace direct fixture-field equality in `dependency_facts_tests/cycle_collection.rs` with `assertions::assert_no_cycles` and make `assert_no_cycles` accept a generic slice.
- **Why:** avoids per-sidecar rule-specific assertions and keeps semantic checks centralized in assertions crate.
- **Alternatives considered:**
  - Copying assertions into the test module itself — rejected because it leaves duplicated semantics across tests.
  - Restricting to exact fixture type — rejected because it reduced helper reusability.

### Assertions crate dependency alignment
- **Chose:** restore the runtime dependency in `crates/app/rs/families/hexarch/crates/assertions/Cargo.toml` and add `use guardrail3_app_rs_family_hexarch as _;` in assertions `lib.rs`.
- **Why:** aligns this assertions crate with other families’ validation contract and removes RS-TEST-03 behavior once validation is run against this crate path.
- **Alternatives considered:**
  - Leaving dependency removed (as a quick compile workaround) — rejected because it caused RS-TEST-03 and broke the expected assertions pattern.

## Architectural Notes
- Kept test sidecars in rule 11–25 scoped directories to only construct fixtures, mutate tests, and invoke assertions modules.
- Assertions modules now own checks for `id`, `title`, `message`, `file`, and result set cardinality semantics.
- Kept changes outside rule ≤10 and `assertions_common` untouched.
