# Arch Rule Split

## Summary

Split the remaining mixed `arch` package rules so each lane now consumes only lane-local public inputs. `RS-ARCH-07` became `RS-ARCH-07A` in file-tree and `RS-ARCH-07B` in config, and `RS-ARCH-08` became `RS-ARCH-08A` in source and `RS-ARCH-08B` in config.

## Decisions made

- Kept rich `G3RsArchCrateNode` facts private to ingestion and fanned out lane-pure public inputs.
  - Why: the earlier extraction was leaking config facts into source/file-tree and vice versa.
- Split `RS-ARCH-07` by the actual predicates it enforced.
  - `dependency_count` moved to config.
  - `module depth`, `sibling dir count`, and `sibling .rs file count` stayed in file-tree.
- Split `RS-ARCH-08` by what it actually checks.
  - source owns facade export gating
  - config owns the `[features]` contract required to support those exports
- Used derived activation for `RS-ARCH-08B` instead of letting the config rule parse source itself.
  - Why: the config rule still checks only config contents, while ingestion decides whether the rule is applicable.
- Fixed the stale `arch` README drift while touching the packages.
  - Why: `g3rs-arch-source-checks` was still describing the `code` family.
