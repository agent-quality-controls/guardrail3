## What was built this session

### The pipeline architecture

```
crawl(workspace_root)
  → G3RsWorkspaceCrawl (neutral filesystem snapshot)
    → ingest(&crawl)
      → G3Rs{Family}ConfigChecksInput (typed parsed data)
        → check(&input)
          → Vec<G3CheckResult>
```

Each family has three packages:
- `g3rs-{family}-types` — the shared input struct (rel paths + parser output types)
- `g3rs-{family}-config-checks` — check rules (one `check()` function, returns `Vec<G3CheckResult>`)
- `g3rs-{family}-config-ingestion` — crawl → select files → parse → return checks input

The dependency chain is clean:
```
parser crates (standalone)
    ↓
g3rs-{family}-types (depends only on parsers)
    ↓              ↓
  checks        ingestion (neither depends on the other)
```

1. **Workspace crawl** — single-root gitignore (rewrote to use ignore::WalkBuilder with ancestor+nested gitignore + targeted recovery)
2. **RS-CARGO-CONFIG-06** — was rejecting `"forbid"` as "weakened" for disallowed_macros

4. **RS-CARGO-CONFIG-02** — required-allow validation was no-op (is_weaker("allow", X) always false)
5. **RS-CARGO-CONFIG-02** — had to revert bidirectional check back to weakening-only (guardrails enforce minimums, stronger is fine)
6. **missing_debug_implementations** — expected level was "warn" but every package uses "deny"
7. **Toolchain checks** — two split input types unified into one
8. **Garde config checks** — two split input types unified into one
9. **Deps config checks** — two split input types unified into one
10. **is_publishable** — workspace-only roots (no [package]) were treated as publishable
11. **Semver validation** — was using weak `has_major_minor` instead of `semver::Version::parse()`
12. **publish = []** — empty registry list not treated as unpublishable
13. **Parser Value re-export** — release-plz and cliff parsers missing `pub use toml::Value`

### Design decisions made

1. **Crawl uses ignore::WalkBuilder + targeted recovery** — not a full walkdir of everything. Phase 1 walks with correct gitignore. Phase 2 recovers specific config files from ignored space. Banned dirs (target, node_modules) excluded from both phases.

2. **Ingestion returns checks input type directly** — not a custom result struct. `ingest(&crawl) -> Result<G3Rs{Family}ConfigChecksInput, Error>`.

3. **Checks enforce minimums, not exact values** — `is_weaker(expected, actual)` only fires when actual is weaker. Stronger is always fine. Required-allow lints (like redundant_pub_crate) use exact match instead.

4. **Optional files produce None, not error** — clippy config in garde, Cargo.toml in toolchain, release-plz/cliff in release. When optional files are missing or malformed, the corresponding fields are None and those checks skip.

5. **Required files produce error** — Cargo.toml in cargo/deny/garde/release ingestion, rust-toolchain.toml in toolchain ingestion. Missing required files fail the ingestion.

6. **Recovery list matches the old app exactly** — same RECOVER_EXACT, RECOVER_PREFIX, path-specific rules, banned dirs.

7. **No global/exclude gitignore semantics** — validation must be machine-independent. Only checked-in .gitignore files affect ignore state.

8. **Shared types packages have no feature gates** — pure data types, always public.

9. **Every checks package: one input type, one check function** — unified from the old split patterns (toolchain had 2, garde had 2, deps had 2).

## What's NOT done

### deps-config-ingestion

The last config family without an ingestion package. This is the complex one:
- Per-crate (returns `Vec<Input>` not a single input)
- Needs workspace member resolution (parse [workspace].members, match against crawl entries)
- Needs local path dependency resolution (parse each member's [dependencies] path entries)
- Needs `guardrail3.toml` parsing — but the current `g3rs-deps-types` uses `GuardrailConfig` from the OLD APP's `guardrail3-domain-config`

**Critical finding:** The `g3rs-deps-types` package depends on `apps/guardrail3/crates/domain/config` for `GuardrailConfig`. This app dependency needs to be replaced with a standalone parser. A `guardrail3-toml-parser` crate needs to be created (same pattern as all other parsers) that outputs a `Guardrail3Toml` type. Then `g3rs-deps-types` uses that instead of the app domain type.

Note: `guardrail3-rs-toml-parser` already exists but parses a DIFFERENT file (`guardrail3-rs.toml` — the per-workspace Rust policy file). The repo-level `guardrail3.toml` is a different schema.

## Process and patterns

### How to build a new ingestion package

1. Read the checks types package to understand the input struct
2. Create the ingestion package following the deny-config-ingestion pattern (the cleanest reference)
3. Structure: facade, types (error only), runtime (fs, select, parse, ingest, run), assertions, tests
4. The `ingest()` function: select files from crawl → parse with parser crates → assemble checks input
5. Return the checks input type from `g3rs-{family}-types` (NOT from checks facade)
6. Required files → error. Optional files → None.
7. Tests: happy path, missing file, malformed file, gitignored-but-recovered (verify ignore_state), dot-variant preference (verify parsed content)
8. Run on real repo files and verify output manually
9. Run adversarial test attacks

### How to build a new checks package

1. Follow g3rs-release-config-checks as reference (the most recent, cleanest)
2. One module per check rule (mod.rs + rule.rs + rule_tests/)
3. Golden test + at least one failure test per rule
4. Support module with shared helpers (is_publishable, crate_name, error/warn/info builders)
5. Feature gate: runtime uses "checks", types have no gate
6. `[package.metadata.guardrail3] shared = true` on facade

### How to build a new parser

1. Follow cliff-toml-parser as reference
2. Structure: facade, types (the struct), runtime (parser, error, fs), assertions
3. Types: `#[derive(Debug, Clone, Deserialize)]`, `#[non_exhaustive]`, `#[serde(flatten)]` for extra
4. Re-export `toml::Value` through facade (for extra fields)
5. Tests: parse real file from repo, empty string, unknown keys in extra, from_path, error cases

### Adversarial testing process

1. Write the code
2. Run on real repo files (not just synthetic fixtures)
3. Manually verify every output is correct
4. Launch test-attack agents to find bugs
5. Fix everything found
6. Run again until clean
