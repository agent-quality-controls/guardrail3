# Per-app type profiles for TypeScript

**Date:** 2026-03-17 15:56
**Scope:** domain/config/types.rs, domain/report.rs, ts/validate/mod.rs, ts/validate/ts_arch_checks.rs, main.rs, commands/validate.rs, commands/init.rs, help_gen.rs, adversarial_categories.rs

## Summary
Added per-app type configuration for TypeScript monorepos. Each app declares its type (service/content/library) which determines which check categories apply. Per-app check overrides provide escape hatch.

## Context & Problem
In a monorepo with heterogeneous TS apps (landing page + admin panel + shared library), different apps need different checks. A landing page needs content/accessibility/SEO checks (future), an admin panel needs hex arch enforcement. The previous global `[typescript.checks]` applied the same categories to all apps.

## Decisions Made

### Profile-driven, not per-category toggles
- **Chose:** `type = "service"` determines defaults, explicit `checks.*` overrides for edge cases
- **Why:** One field per app covers 95% of cases. Config doesn't explode with N apps × M categories.
- **Alternatives considered:**
  - Per-app check category map — rejected because it's verbose and semantically wrong (you're describing what the app IS, not manually toggling booleans)
  - Separate guardrail3.toml per app — rejected because it fragments shared config

### App type determines architecture gating
- **Chose:** Service apps get T-ARCH checks, Content/Library skip them
- **Why:** Content sites don't use hex arch (no modules/domain/adapters). Libraries are packages, not apps.

### Default type is Service
- **Chose:** Unconfigured apps treated as Service (strictest)
- **Why:** Fail-safe — if you forget to configure, you get all checks. Opt out explicitly.

## Config shape
```toml
[typescript.apps.landing]
type = "content"

[typescript.apps.admin]
type = "service"

[typescript.apps.admin.checks]
architecture = false   # override: service but skip hex arch
```
