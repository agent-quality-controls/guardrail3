## What This Handoff Got Right

These decisions remained correct and were later implemented:
- content checks belong in packages under `packages/`
- app families own discovery, routing, and parse-failure handling
- content-check packages receive parsed file types, not raw TOML
- package inputs should be concrete and non-optional
- `G3CheckResult`, `G3Severity`, and `G3Profile` are the shared package-level types
- only content checks move; filetree/coverage/shadowing logic stays in the app

## What Was Outdated In The Original Note

The original handoff predated several decisions and renames:
- parser package names later became `*-parser`
- `toolchain`, `fmt`, `clippy`, `deny`, and `cargo` were later extracted in phases
- the unresolved debate about “full parsed types vs pre-extracted fields” is resolved:
  full parsed file types cross the package boundary
- malformed typed parses are structurally owned by the app family, not by content packages
- cargo was later corrected to a one-file content-check package instead of a workspace-bag design

## Durable Architectural Rule

The durable rule that survived this handoff is:

```text
app family orchestrator
  -> discovers authoritative files
  -> parses files
  -> emits structural/parse failures
  -> calls content-check package with parsed file types
  -> package validates content only
```

Anything that violates that rule should be treated as drift.
