# Session 3 handoff — family extraction design

**Date:** 2026-04-04 15:06

## The problem we're trying to solve

guardrail3 has 15 "families" of rules that validate Rust project structure
and configuration. Each family is an internal crate inside the app. They're
entangled: they share types, cross-read each other's config files, duplicate
parsing logic, and mix two fundamentally different concerns (file placement
vs file content validation).

We want to extract the content validation rules into independent packages
under `packages/` so that:
- Each package is independently testable without the full app
- The type system enforces what data each family can access
- Families don't need to know about each other
- New families can be added without touching app internals
- Content validation is reusable outside guardrail3

## Design decisions (resolved)

### 1. App parses everything
The app reads files, calls parser crates, passes typed structs to packages.
Packages never receive raw strings. Packages never call parse().

### 2. One result type
`GrdzCheckResult` from check-types is THE result type everywhere. The app's
internal `CheckResult` needs to either become a re-export of `GrdzCheckResult`
or be replaced entirely. This is a prerequisite for extraction.

### 3. Non-optional typed inputs
If a file doesn't exist or failed to parse, the content checks package
doesn't get called. The app handles file-missing errors and parse errors.
Content checker inputs have zero Options for "does the file exist" — the
file always exists when the package is called. Fields ARE optional when the
underlying data is legitimately optional (e.g., Cargo.toml may not declare
rust-version — that's a valid state, not a missing file).

### 4. Parse failures are upstream's problem
The app emits parse error results. Content checkers assume valid input.

### 5. Packages check internal correctness only
No knowledge of what was skipped upstream, what other families exist, or
what other roots look like. Given parsed data, validate it, return results.

### 6. Split by concern, not by file
Each family checks its own concerns across multiple file types:
- Garde checks clippy.toml for garde-specific bans (garde's concern)
- Clippy does NOT check for garde bans (not clippy's concern)
- Each family receives whatever parsed files it needs for ITS concerns

### 7. Profile is Application or Library
Determined from guardrail3.toml. Affects which policy variant to apply.

### 8. Content checks vs filetree checks
Each family has both:
- Filetree: does the file exist, is it in the right place, coverage
- Content: given a file, is its content correct

Only content checks get extracted to packages. Filetree checks stay in
the app's family runtime. The app handles discovery, the package handles
validation.

## Design decisions (unresolved)

### A. What does the input struct look like exactly?
We decided: parsed types, non-optional, minimal necessary fields. But the
exact struct design keeps flip-flopping. Key tension:

Should the input contain the full parsed config type from the parser crate
(e.g., `clippy_toml::ClippyConfig`) or should it contain pre-extracted
fields (e.g., `channel: String, components: Vec<String>`)?

Full parsed type:
- Pro: package can access any field, no coupling on which fields matter
- Con: package depends on the parser crate type, and it's pub fields
  (or the package navigates the type)

Pre-extracted fields:
- Pro: input is minimal, no dependency on parser crate types
- Con: the app needs to know which fields to extract, adding coupling.
  Also if a rule needs a new field, the input struct must change.

NOT RESOLVED. The toolchain plan flip-flopped between toml::Value, 
full parsed types, and pre-extracted fields.

### B. Per-family input types — who defines them?
Each family defines its own input struct (GrdzClippyChecksInput,
GrdzToolchainChecksInput, etc.). These live in the checks package.
The app imports them to construct inputs.

This means the app depends on every checks package (for the input types).
That's fine — the app is the orchestrator.

### C. How does the app transition from CheckResult to GrdzCheckResult?
The app currently uses `guardrail3_domain_report::CheckResult` everywhere.
The checks packages return `GrdzCheckResult`. Options:
- Big-bang: replace CheckResult with GrdzCheckResult everywhere at once
- Gradual: conversion functions during migration
- Re-export: make domain/report re-export GrdzCheckResult as CheckResult

NOT RESOLVED. Affects every extraction.

### D. How do cross-family concerns work?
Example: garde needs to check that clippy.toml has garde-specific bans.
The garde checks package receives a parsed `ClippyConfig`. This means:
- Garde depends on `clippy-toml` parser crate (for the type)
- The app passes the parsed clippy.toml to garde
- Garde reads the disallowed-methods/types to verify its bans exist

This is clean — garde depends on a parser crate, not on the clippy
checks package. But it means multiple packages depend on the same
parser crate types.

### E. What about rules that use toml::Value navigation?
Many rules (especially toolchain, deny) navigate raw `toml::Value` with
.get("key") chains. If the app parses to typed structs, the rules need
rewriting to use typed field access. That's work — but it's the right
thing since the typed parser exists.

### F. Global vs per-workspace families
Per-workspace families (toolchain, clippy, deny, cargo, hexarch, deps,
garde, release) get called once per workspace root.

Global families (topology, arch, fmt, code, test, hooks-shared, hooks-rs)
get called once for the entire project.

The extraction pattern differs. Per-workspace packages receive data for
ONE root. Global packages receive data for ALL roots.

Only per-workspace content checks are designed so far. Global families
are deferred.
